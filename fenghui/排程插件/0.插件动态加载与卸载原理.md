# 插件动态加载与卸载原理

## 1. 先给结论

这个插件管理模块不是把插件 DLL 直接加载自 `Plugins/{pluginId}` 目录，而是：

1. 先扫描插件目录，读取每个插件的 `plugin.json`
2. 启用时把整个插件目录复制到 `Plugins/.runtime/{pluginId}/{时间戳-guid}` 影子目录
3. 再从这个影子目录用独立的 `AssemblyLoadContext` 加载插件
4. 禁用时释放插件子容器、调用生命周期停止方法、卸载 `AssemblyLoadContext`

所以：

- 运行中的插件代码，实际执行的是影子目录里那一份 DLL
- 直接替换 `Plugins/{pluginId}` 下的 DLL，不会让当前已启用插件“立刻生效”
- 新 DLL 要生效，至少需要一次“禁用 -> 启用”
- 当前实现没有文件监听、没有热更新、没有自动重载逻辑

## 3. 这个模块在管理什么

### 3.1 三层对象

这个模块同时管理三层信息：

1. 磁盘文件
   - 插件原始目录：`Plugins/{pluginId}`
   - 运行时影子目录：`Plugins/.runtime/{pluginId}/{version-instance}`
2. 数据库记录
   - 表：`system_plugin`
   - 实体：`SysPlugin`
3. 进程内运行时状态
   - 字典：`PluginManager._plugins`
   - 值对象：`PluginEntry`

### 3.2 运行时状态

状态定义在 `PluginState`：

- `Installed = 0`
- `Enabled = 1`
- `Disabled = 2`
- `Error = 3`

含义是：

- `Installed`：已安装，但尚未加载到进程
- `Enabled`：程序集已加载，插件子容器已构建，必要时后台任务已启动
- `Disabled`：程序集已卸载，文件保留，可再次启用
- `Error`：加载或运行失败，需要人工干预

## 4. 插件启动时是如何被发现和加载的

### 4.1 启动时机

`UsePluginStartupLoader()` 在 `ApplicationStarted` 后注册异步扫描：

- 读取 `PluginOptions`
- 解析 `ScanPath`
- 创建插件目录
- 调用 `PluginManager.ScanAsync(pluginsDir)`

这意味着插件扫描发生在服务启动完成之后，而不是在 DI 容器构建阶段。

### 4.2 扫描流程

核心代码：`PluginManager.ScanAsync()`，见 `PluginManager.cs:96-172`

流程如下：

1. 如果插件目录不存在，直接跳过
2. 清理历史残留的 `.runtime` 影子目录
3. 查询数据库中状态为 `Enabled` 的插件 ID
4. 枚举 `Plugins` 根目录下的每个子目录
5. 读取 `plugin.json`
6. 反序列化成 `PluginManifest`
7. 放入 `_plugins` 内存字典，初始状态设为 `Installed`
8. 如果配置 `AutoEnable=true` 且数据库里该插件之前是 `Enabled`，则自动调用 `EnableAsync()`

这里可以看出两个关键点：

- 扫描负责“发现插件”和“恢复已启用状态”
- 扫描本身不会无条件启用所有插件，只会恢复数据库中原本启用过的插件

## 5. 插件安装流程

核心代码：`PluginManager.InstallAsync()`，见 `PluginManager.cs:184-285`

### 5.1 安装时做了什么

1. 将上传的 ZIP 流转成可 Seek 的流
2. 打开 ZIP，强制要求其中存在 `plugin.json`
3. 解析 `plugin.json` 为 `PluginManifest`
4. 验证主程序集 `manifest.Assembly` 在 ZIP 中存在
5. 对插件 ID 加锁，防止并发安装同一插件
6. 检查内存字典和数据库中是否已有同 ID 插件
7. 解压到 `Plugins/{pluginId}`
8. 写入数据库 `system_plugin`
9. 在 `_plugins` 中创建 `PluginEntry`，状态为 `Installed`

### 5.2 重要限制

这个系统当前不支持“同 ID 覆盖安装升级”。

代码在 `PluginManager.cs:220-231` 明确限制：

- 内存中已存在同 ID，拒绝安装
- 数据库中已存在同 ID，拒绝安装

所以如果走正式安装流程升级插件，当前模型更接近：

- 先禁用/卸载旧插件
- 再安装新 ZIP

## 6. 启用流程是整个动态加载的核心

核心代码：`PluginManager.EnableAsync()` 与 `EnableInternalAsync()`，见：

- `PluginManager.cs:297-352`
- `PluginManager.cs:550-683`

### 6.1 启用总体流程

`EnableAsync(pluginId)` 做的是控制层逻辑：

1. 按插件 ID 加锁
2. 检查插件是否存在
3. 检查状态是否允许启用
4. 调用 `EnableInternalAsync(entry)`
5. 启用成功后把数据库状态更新为 `Enabled`

真正的动态加载发生在 `EnableInternalAsync()`。

### 6.2 第一步：清理旧运行时目录

代码：`PluginManager.cs:554`

每次启用前都会先执行：

- `CleanupRuntimeDirectory(entry, deleteAllVersions: true)`

目的：

- 删除上一次运行残留的影子目录
- 确保这次启用从一份新的影子副本启动

### 6.3 第二步：创建影子目录

代码：`PluginManager.cs:556-557`、`960-973`

系统不会直接从 `Plugins/{pluginId}` 加载 DLL，而是先：

1. 在 `Plugins/.runtime/{pluginId}/` 下创建一个唯一目录
2. 将整个源插件目录完整复制进去
3. 把 `entry.RuntimeDirectory` 指向这个新目录

这就是“影子复制”。

这个设计的直接收益是：

- 原始插件目录通常不被运行中的程序集直接锁定
- 更容易在不停服务的情况下替换源目录中的插件文件
- 每次启用都能从一个干净目录重新加载

### 6.4 第三步：创建独立的 AssemblyLoadContext

代码：`PluginManager.cs:559-565`

```csharp
var mainAssemblyPath = Path.Combine(runtimeDir, manifest.Assembly);
var loadContext = new PluginLoadContext(mainAssemblyPath);
assembly = loadContext.LoadFromPluginDir(manifest.Assembly);
```

这里的关键不是 `LoadFromAssemblyPath` 本身，而是：

- 每个插件都拥有自己的 `PluginLoadContext`
- `PluginLoadContext` 是 `isCollectible: true`

这表示它是一个“可回收的独立程序集加载上下文”，理论上可以在禁用时被卸载。

### 6.5 第四步：程序集解析策略

核心代码：`PluginLoadContext.Load()`，见 `PluginLoadContext.cs`

它的加载优先级是：

1. 先复用默认上下文中已经加载的共享程序集
   - 例如 `Plugin.Contract`、Furion、宿主框架程序集
2. 如果宿主输出目录里有同名程序集，交给默认上下文加载
3. 否则通过 `AssemblyDependencyResolver` 从插件本地目录解析
4. 最后再尝试回退到默认上下文

这套策略的含义很重要：

- 插件自己的独立依赖，可以从插件目录单独加载
- 与宿主共享的抽象和框架程序集，尽量只保留一份类型身份
- 避免 `Plugin.Contract` 被加载成两份，导致接口类型不相等

换句话说，这不是“完全隔离”，而是“共享框架 + 隔离插件私有依赖”的折中方案。

### 6.6 第五步：找到启动类并实例化

代码：`PluginManager.cs:576-615`

系统会：

1. 根据 `plugin.json` 中的 `startupClass` 取类型
2. 校验它必须实现 `IPlugin`
3. 用 `Activator.CreateInstance()` 实例化
4. 把实例保存到 `entry.Instance`

插件最基本的入口契约是 `IPlugin`，它只描述身份信息：

- `Id`
- `Name`
- `Version`
- `Type`

### 6.7 第六步：构建插件子容器

代码：`PluginManager.cs:625-648`

系统这里没有把宿主根容器整个传给插件，而是：

1. 新建一个 `ServiceCollection`
2. 只放入日志相关基础服务
3. 反射调用插件启动类的 `ConfigureServices(IServiceCollection)`
4. 再单独 `BuildServiceProvider()`

这意味着：

- 插件拥有自己的子容器
- 插件需要的内部服务，必须由插件自己注册
- 宿主的大量业务服务默认并不会直接注入给插件

这是当前实现里非常重要的一条边界。

### 6.8 第七步：校验 operation 与执行器

代码：`PluginManager.cs:650-652`、`854-912`

系统会把两类声明做一致性校验：

1. `plugin.json` 的 `operations`
2. 如果插件实现了 `IPluginDescriptor`，其 `Operations`

然后再检查：

- 是否注册了 `IPluginExecutor`
- 每个 operation 是否有且仅有一个执行器可以处理

如果插件声明了 operation 但没有执行器，或者一个 operation 有多个执行器，启用会失败。

### 6.9 第八步：启动生命周期

代码：`PluginManager.cs:660-676`

如果插件实现了 `IPluginLifecycle`，启用时会调用：

- `StartAsync(PluginContext context)`

`PluginContext` 里目前会传：

- `PluginDirectory`
- `Configuration`

因此，长生命周期插件可以在这里：

- 启动后台线程
- 建立长连接
- 读取插件配置

而无状态算法类插件通常只实现 `IPlugin` + `IPluginExecutor`，不需要实现 `IPluginLifecycle`。

### 6.10 第九步：状态切换为 Enabled

代码：`PluginManager.cs:678-682`

如果以上流程全部成功，内存状态置为 `Enabled`，随后 `EnableAsync()` 会把数据库状态也更新成 `Enabled`。

## 7. 插件调用流程

这一部分和“动态加载”是两条链，但实际使用时它们是连起来的。

### 7.1 调用入口

API 入口：

- `PluginInvokeAppService.Execute()`  
  文件：`PluginInvokeAppService.cs`

调用门面：

- `PluginInvokeFacadeService.InvokeAsync()`  
  文件：`PluginInvokeFacadeService.cs`

### 7.2 当前版本的调用数据形态

当前代码里有一个实现细节：

- 对外 API DTO：`Plugin.Entity.Dto.PluginInvokeInput`
  - `Context` / `Payload` 是 `JObject`
- 插件契约 DTO：`Plugin.Contract.Dto.PluginInvokeRequest`
  - `Context` / `Payload` 是 `string`

所以门面层会做一次转换：

1. 接口先收 `PluginInvokeInput`
2. `PluginInvokeFacadeService` 把 `Payload` / `Context` 转成字符串
3. 再组装成 `PluginInvokeRequest`
4. 传给插件执行器

代码在 `PluginInvokeFacadeService.cs:37-58`。

### 7.3 调用执行链

调用时序如下：

1. `PluginInvokeAppService.Execute()` 收到请求
2. `PluginInvokeFacadeService` 做基础校验和日志
3. `PluginRuntimeRegistry.GetRequiredEnabled(pluginId)` 校验插件必须是 `Enabled`
4. `PluginExecutorFactory.Create(pluginId, operation)` 找到唯一执行器
5. 构造 `PluginExecutionContext`
6. 调用 `IPluginExecutor.ExecuteAsync(request, context, cancellationToken)`
7. 返回 `PluginInvokeResult`

这里说明一个本质问题：

- 插件并不是通过反射直接调用任意类方法
- 宿主只通过 `IPluginExecutor` 抽象与插件交互

这使得调用路径稳定、边界清晰，也更利于做校验和审计。

## 8. 禁用流程

核心代码：`PluginManager.DisableAsync()` 与 `DisableInternalAsync()`，见：

- `PluginManager.cs:364-410`
- `PluginManager.cs:688-735`

### 8.1 禁用时做了什么

1. 按插件 ID 加锁
2. 如果插件实现了 `IPluginLifecycle`，调用 `StopAsync(context)`
3. 释放插件子容器
4. 清空 `entry.ServiceProvider`
5. 清空 `entry.Instance`
6. 卸载 `entry.LoadContext`
7. 触发 GC 和 Finalizer，等待上下文回收
8. 删除影子目录
9. 状态更新为 `Disabled`

### 8.2 为什么要等待卸载

代码：`WaitForUnload()`，见 `PluginManager.cs:1084-1100`

它会循环：

- `GC.Collect()`
- `GC.WaitForPendingFinalizers()`
- 再 `GC.Collect()`

原因是：

- `AssemblyLoadContext.Unload()` 只是发起卸载
- 真正回收要等没有任何对象再引用这个上下文里的类型或实例

如果仍然有引用，日志会提示：

- 该 `AssemblyLoadContext` 未在预期时间内完全卸载

这通常意味着插件自己还有残留：

- 后台线程没停
- 定时器没释放
- 静态事件未取消订阅
- 宿主还持有插件对象引用

## 9. 卸载流程

核心代码：`PluginManager.UninstallAsync()`，见 `PluginManager.cs:423-463`

卸载逻辑是：

1. 如果插件当前是 `Enabled`，先执行 `DisableInternalAsync()`
2. 删除 `.runtime` 影子目录
3. 删除原始插件目录 `Plugins/{pluginId}`
4. 删除数据库记录
5. 从 `_plugins` 内存字典移除

所以：

- 禁用只是“从进程退出，但文件保留”
- 卸载才是“从系统中彻底删掉”

## 10. 动态加载/卸载原理的本质总结

这个模块的动态插件机制，核心其实只有三点：

### 10.1 用 `plugin.json` 做插件发现

宿主不需要预先编译时知道每个插件，只要目录下存在合法 `plugin.json`，就能发现插件。

### 10.2 用 `AssemblyLoadContext` 做运行时隔离

每个启用中的插件都有自己的 `PluginLoadContext`，这样可以：

- 在运行时单独加载
- 在禁用时尝试单独卸载

### 10.3 用“影子复制”避免直接锁住原始插件目录

真正加载的位置是 `.runtime` 目录，而不是源目录。  
这让“停服务前先替换源插件文件”变得更容易。

## 11. 不重启服务，动态替换插件 DLL 后会不会立即生效

## 11.1 结论

不会立即生效。

更准确地说：

- **当前已经启用并正在运行的插件，不会因为你替换了源目录 DLL 而立刻切换到新代码**
- **新 DLL 只有在下一次重新启用插件时才会被复制到新的影子目录并加载**

## 11.2 为什么不会立即生效

原因来自三层：

### 原因 1：当前运行的是影子目录里的 DLL

启用时执行的是：

- `CreateRuntimeLoadDirectory()`
- `LoadFromPluginDir()` 加载的是 `RuntimeDirectory`

也就是说，进程里已经加载的是 `.runtime/...` 下那份文件。  
你替换 `Plugins/{pluginId}` 下的源 DLL，不会影响已经加载到内存中的程序集。

### 原因 2：程序集一旦被 CLR 加载，不会因为文件变化自动重读

.NET 运行时不会监控 DLL 文件变化，更不会自动把已加载程序集替换成新版本。

### 原因 3：当前实现没有热更新监听机制

代码中没有：

- `FileSystemWatcher`
- 插件目录变更监听
- 自动禁用/重启
- 热替换或热补丁逻辑

所以文件变化本身不会触发任何运行时动作。

## 11.3 那么在“不重启服务”的前提下，怎么让新 DLL 生效

当前实现下，推荐方式是：

1. 调用插件禁用接口
2. 替换 `Plugins/{pluginId}` 下的插件文件
3. 再调用插件启用接口

原因是：

- 禁用时旧的 `AssemblyLoadContext` 会尝试卸载
- 再次启用时会从当前源目录重新复制到新的 `.runtime` 目录
- 这时新 DLL 才会被加载

也就是说：

- **不重启整个服务可以**
- **但至少要重启这个插件实例**

## 11.4 直接替换源 DLL 的可行性

由于当前运行的是影子副本，通常情况下：

- 替换 `Plugins/{pluginId}` 下源文件，比直接替换运行中的 `.runtime` 目录更可行

但要注意：

- 如果插件包含本地库或复杂依赖，只替换单个 DLL 可能造成版本不一致
- 更安全的方式是替换整个插件目录内容，或走“卸载旧包 -> 安装新包”

## 11.5 如果插件禁用后仍然没有完全卸载，会怎样

这时有两层结果：

1. 旧代码不会“自动变成新代码”
2. 新启用时大概率仍会加载一份新的影子目录副本，但旧上下文可能残留在内存里

代码里已经考虑了这种情况，会打印日志警告：

- `AssemblyLoadContext 未在预期时间内完全卸载`

这说明插件作者需要重点检查：

- `StopAsync()` 是否真正停止后台线程
- 是否释放了定时器
- 是否取消了静态事件订阅
- 是否有全局缓存还持有插件对象

## 12. 这个模块对“插件热更新”目前的实际能力边界

可以做到的：

- 不重启整个宿主服务
- 动态安装插件
- 动态启用插件
- 动态禁用插件
- 动态卸载插件
- 通过禁用再启用，加载替换后的新 DLL

做不到的：

- 运行中的插件代码自动热替换
- 只替换 DLL 文件就立刻切到新逻辑
- 无感知在线升级
- 自动监听插件目录并自动重载

所以这个模块目前更准确的定位是：

- **支持插件级别的动态加载/卸载**
- **不支持真正意义上的热更新/热替换**

## 13. 如果要把“替换 DLL 后自动生效”做成产品能力，需要补什么

如果后面要增强，可以考虑补这些能力：

### 13.1 文件监听

监听 `Plugins/{pluginId}` 目录变化，发现 DLL 或 `plugin.json` 变化后触发重载流程。

### 13.2 显式“重载插件”接口

新增一个 `ReloadAsync(pluginId)`：

1. `DisableAsync(pluginId)`
2. 校验新文件
3. `EnableAsync(pluginId)`

这比让运维手工“禁用 -> 替换 -> 启用”更安全。

### 13.3 升级事务化

把“停旧版本、校验新版本、启新版本、失败回滚”做成一个原子过程。

### 13.4 卸载约束校验

在禁用/重载前检查：

- 是否仍有线程活跃
- 是否仍有事件订阅
- 是否仍有未释放资源

### 13.5 版本并存或蓝绿切换

如果未来对高可用要求更高，可以考虑：

- 同一插件多版本并存
- 新旧版本切流
- 验证通过后再切换

当前实现还没有做到这一步。

## 14. 学习这套代码时建议重点抓住的 5 个点

1. `PluginApplicationBuilderExtensions.UsePluginStartupLoader()`
   - 看宿主什么时候开始扫描插件
2. `PluginManager.ScanAsync()`
   - 看插件如何被发现、如何恢复已启用状态
3. `PluginManager.EnableInternalAsync()`
   - 这是动态加载的主战场
4. `PluginLoadContext`
   - 看程序集隔离和依赖解析策略
5. `PluginManager.DisableInternalAsync()`
   - 看插件为什么能被“卸载”

## 15. 最终一句话总结

这套插件系统的设计本质是：

**用 `plugin.json` 发现插件，用影子目录复制插件文件，用独立 `AssemblyLoadContext` 动态加载插件，用禁用流程释放子容器和生命周期资源，再尝试卸载上下文。**

因此：

**不重启服务可以替换插件文件，但替换后不会立即生效；必须至少重新启用该插件，新的 DLL 才会被真正加载。**
