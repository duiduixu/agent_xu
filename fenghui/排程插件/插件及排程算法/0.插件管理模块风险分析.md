  对你关心的问题，直接结论

  - 安装多个插件会不会出问题
    会有两类问题。第一类是性能线性增长：启动扫描、启用影子复制、内存占用都会叠加。第二类是依赖冲突：虽然每个插件有独立 AssemblyLoadContext，但对默认上下文程序集做了优先复用，所以不是完全隔离，多插件版本不一致时有概率互相影响。

  - 频繁安装、启用、卸载会不会出问题
    会。最现实的是磁盘 I/O 激增、.runtime 目录残留、上下文卸载不干净、状态分裂，以及管理操作和执行操作竞态。尤其是“插件正在执行时被禁用/卸载”这一类，当前代码没有保护。

  - 会不会产生内存泄露
    有这个风险，而且不是理论风险。主要来源是：
      1. 插件没有完全释放自身线程/定时器/事件订阅，导致 ALC 不能卸载。
      2. 插件 Scoped 服务被根容器长持有。
      3. 影子目录和上下文清理失败只有 warning。
      4. 锁字典不回收。

  生产前最该补的 8 件事

  - 给插件执行链增加“活动调用计数”或读写锁。执行时加读锁，禁用/卸载时加写锁，确保没有正在跑的请求。
  - 改插件 DI 模型。每次调用创建 IServiceScope，执行器和依赖从 scope 解析，不要直接从根 ServiceProvider 拿。
  - 给 ReloadAsync(pluginId) 做成正式能力，流程固定为“停旧 -> 校验新包 -> 启新 -> 失败回滚”。
  - 把启动扫描改成受控就绪流程。至少要么阻塞 readiness，要么把插件未加载完成状态显式暴露出去。
  - 为 .runtime 和未卸载 ALC 做监控。建议采集：启用耗时、影子目录数量、每插件残留目录数、卸载失败次数、进程私有内存。
  - 明确插件依赖规则。共享哪些程序集、禁止插件自带哪些程序集、是否强制统一版本，需要文档和构建校验。
  - 给安装/卸载做一致性修复工具。比如“重建 DB 记录”“清理孤儿目录”“清理孤儿 runtime 目录”。
  - 给插件开发规范加硬约束。必须停止后台线程、释放 Timer、取消静态事件、避免全局单例缓存插件对象。

  额外架构风险

  - 如果未来要做多实例部署，这套实现默认是单机思维：插件文件在本地磁盘，状态在数据库，没做分布式锁，也没做多节点文件同步。单节点没问题，多节点会很快出现某节点已安装、某节点未安装，或状态不一致的问题。

  建议的压测/演练场景

  - 10 个插件同时安装并启用，测启动时间、内存、.runtime 目录大小。
  - 单插件连续 100 次启用/禁用，观察 ALC 是否回收、影子目录是否残留。
  - 插件执行中并发触发禁用/卸载，验证是否出现异常和悬挂上下文。
  - 2 个插件分别依赖同名不同版本基础库，验证是否发生版本串用。
  - 大 payload 高频调用插件接口，观察 GC、分配率和耗时抖动。
