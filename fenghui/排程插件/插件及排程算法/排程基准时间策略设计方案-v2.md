# 排程基准时间策略设计方案 v2

## 1. 需求重新定义

### 1.1 v1 方案的偏差

v1 方案中，模式 B 的实现是将 `PlanStartTime` 转换为 `EarliestStartMinutes`（最早可开始时间下界），本质上仍是"不得早于该时间"的单边约束。求解器在满足这个下界后，仍会自由地把工单尽量往前排以减少总工期，并不能保证工单"在用户指定的时间开始"。

### 1.2 真实需求

与用户深度沟通后，确认需求如下：

> 用户希望直接指定工单的**目标开始时间**，算法负责计算结束时间。目标时间是"最高优先级意图"，不是绝对硬约束——当资源冲突导致无法精确满足时，系统**不应直接延期或放弃该工单**，而是：
> 1. 仍然安排该工单（尽量贴近目标时间）
> 2. 在响应中输出冲突警告，说明偏移量，并给出建议时间

此外，用户希望通过一个 `StartTimeStrictness`（开始时间严格度，0~100）滑块来控制目标时间的优先级强度：
- `0`：完全忽略 `PlanStartTime`，纯优化模式（等同于旧模式 A）
- `1~99`：软约束模式，严格度越高，偏移惩罚权重越大
- `100`：极高权重模式，求解器会竭尽所能地贴近目标时间，并对任何偏移给出冲突警告

## 2. 核心设计思路

### 2.1 目标时间 vs 最早开始时间的本质区别

| 概念 | 语义 | 实现方式 |
|------|------|---------|
| `EarliestStartMinutes`（v1） | 不得早于此时间 | 硬约束 `start >= earliest` |
| `TargetStartMinutes`（v2） | 希望在此时间开始 | 硬下界 + 软惩罚 + 冲突检测 |

`TargetStartMinutes` 的约束体系分为两层：

**层 1（硬下界）**：`start >= targetStart`
- 语义：工单不会在用户指定时间之前开始（尊重用户意图，不提前）
- 对所有 `strictness > 0` 的情况都生效

**层 2（软惩罚）**：`deviation = start - targetStart`，惩罚 = `deviation × deviationWeight`
- 语义：超过目标时间越多，惩罚越大，求解器会主动避免延误
- `deviationWeight` 由 `StartTimeStrictness` 控制

### 2.2 StartTimeStrictness 的权重映射

现有目标函数中各权重的数量级：
- `TardinessWeight = 20`（每分钟拖期惩罚）
- `ColorBacktrackWeight = 180`（颜色倒退惩罚）

`StartTimeStrictness` 与 `deviationWeight` 的映射关系：

```
deviationWeight = StartTimeStrictness × 4

strictness=0    → weight=0   （无惩罚，忽略目标时间）
strictness=25   → weight=100 （= 5 × TardinessWeight，中等优先）
strictness=50   → weight=200 （= 10 × TardinessWeight，高优先）
strictness=100  → weight=400 （= 20 × TardinessWeight，极高优先）
```

在 `strictness=100` 时，每偏移 1 分钟的惩罚等于 20 分钟的拖期惩罚，求解器会在不违反机器/模具约束的前提下竭力贴近目标时间。

### 2.3 冲突检测与警告

排程完成后，对每个有 `TargetStartMinutes` 的动态工单，检查实际开始时间是否偏离目标：

```
deviation = actualStartMinutes - targetStartMinutes
```

当以下条件成立时，产生冲突警告：
- `deviation > 0`（实际开始晚于目标）
- `strictness >= 警告阈值`（低严格度下小偏移不产生噪音警告）

建议阈值：`strictness >= 30` 时才生成警告，避免低严格度下的信息过载。

冲突警告格式（输出到 `StartTimeConflicts` 字段）：

```
工单 {OrderChildNo} 目标开始时间 {targetTime}，
实际排定 {actualTime}（偏移 {deviation} 分钟）。
建议：将目标开始时间调整为 {actualTime}。
```

## 3. 架构变更详解

### 3.1 第一步：废弃 `ScheduleTimeMode`，新增 `StartTimeStrictness`

v1 方案提出的 `ScheduleTimeMode` 枚举不再需要，用 `StartTimeStrictness` 单一字段替代：

```csharp
// Domain/SchedulingSettings.cs — 修改 record 参数
public sealed record SchedulingSettings(
    int HorizonMinutes,
    int TardinessWeight,
    int SetupWeight,
    int ColorBacktrackWeight,
    int MakespanWeight,
    int StartTimeStrictness = 0)  // ← 新增：0~100，0=忽略目标时间
{
    public static SchedulingSettings Default() =>
        new(
            HorizonMinutes: 48 * 60,
            TardinessWeight: 20,
            SetupWeight: 1,
            ColorBacktrackWeight: 180,
            MakespanWeight: 1,
            StartTimeStrictness: 0);
}
```

**为什么用 `int` 而不是 `double`？** 与现有所有权重字段类型保持一致，宿主 JSON 传 0~100 的整数更直观，CP-SAT 的目标函数只接受整数系数。

### 3.2 第二步：`Job` 新增 `TargetStartMinutes`

```csharp
// Domain/Job.cs — record 末尾新增（带默认值，保持向后兼容）
public sealed record Job(
    // ... 现有所有参数不变 ...
    bool IsStarted = false,
    bool IsFixed = false,
    int? TargetStartMinutes = null);  // ← 新增：相对 scheduleTime 的目标开始分钟数
```

### 3.3 第三步：`SchedulingPreprocessor.Prepare` — Job 构建增加 `TargetStartMinutes` 赋值

```csharp
// 在 3l 步骤构建 Job 之前计算 TargetStartMinutes
int? targetStartMinutes = null;
if (input.Settings.StartTimeStrictness > 0
    && workOrder.PlanStartTime.HasValue
    && !shouldAnchor)
{
    var offset = (workOrder.PlanStartTime.Value - scheduleTime).TotalMinutes;
    if (offset > 0)
    {
        // 目标时间在未来：转换为相对分钟数
        targetStartMinutes = (int)Math.Ceiling(offset);
    }
    // offset <= 0：目标时间已过或等于基准时间，不施加约束（可立即排产）
}

// 超出 Horizon 的目标时间不参与当前排程窗口
if (targetStartMinutes.HasValue && targetStartMinutes.Value >= input.Settings.HorizonMinutes)
{
    unschedulableOrders.Add(new UnschedulableOrder(
        workOrder, $"计划开始时间（{workOrder.PlanStartTime:yyyy-MM-dd HH:mm}）超出排程时间窗口。"));
    continue;
}

var job = new Job(
    // ... 现有参数 ...
    IsStarted(workOrder),
    shouldAnchor,
    targetStartMinutes);  // ← 新增
```

**`scheduleTime` 的取值**：当 `StartTimeStrictness > 0` 时，`scheduleTime` 固定取 `DateTime.Now`（而非最早 PlanStartTime），理由是：
- 目标时间是相对于"现在"的偏移
- 若 `scheduleTime = 最早 PlanStartTime`，则该工单的 `targetStartMinutes = 0`，失去了约束意义

修改 `scheduleTime` 计算逻辑：

```csharp
// scheduleTime 计算（在阶段 2 开头）
var earliestPlanStart = input.WorkOrders
    .Where(x => x.Status == WorkOrderStatus.PendingSchedule && x.PlanStartTime != null)
    .OrderBy(x => x.PlanStartTime)
    .FirstOrDefault()?.PlanStartTime;

DateTime rawScheduleTime;
if (input.Settings.StartTimeStrictness > 0)
{
    // 目标时间模式：以当前时间为基准，PlanStartTime 转换为相对偏移
    rawScheduleTime = DateTime.Now;
}
else
{
    // 纯优化模式：以最早计划开始时间为基准（现有逻辑）
    rawScheduleTime = earliestPlanStart.HasValue && earliestPlanStart.Value > DateTime.Now
        ? earliestPlanStart.Value
        : DateTime.Now;
}

// 截断到分钟
var scheduleTime = new DateTime(
    rawScheduleTime.Year, rawScheduleTime.Month, rawScheduleTime.Day,
    rawScheduleTime.Hour, rawScheduleTime.Minute, 0,
    rawScheduleTime.Kind == DateTimeKind.Unspecified ? DateTimeKind.Local : rawScheduleTime.Kind);
```

### 3.4 第四步：`CpSatOptimizer.Optimize` — 新增目标时间约束和偏差项

改动位于步骤 4b（变量创建后）和步骤 10（目标函数）。

#### 4b 之后：添加目标时间硬下界和偏差变量

```csharp
// 在 jobVars[(job.Id, machineId)] 赋值之后、步骤 6 之前

// 收集所有工单的偏差变量（用于目标函数）
var deviationTerms = new List<LinearExpr>();

foreach (var job in problem.Jobs)
{
    // 仅对有目标时间且严格度 > 0 的工单处理
    if (!job.TargetStartMinutes.HasValue || problem.Settings.StartTimeStrictness <= 0)
        continue;

    var target = job.TargetStartMinutes.Value;
    var deviationWeight = problem.Settings.StartTimeStrictness * 4;

    foreach (var machineId in job.EligibleMachines)
    {
        var vars = jobVars[(job.Id, machineId)];

        // 硬下界：不得早于目标时间开始（只有分配到该机器时生效）
        model.Add(vars.Start >= target).OnlyEnforceIf(vars.Present);

        // 偏差变量：deviation = start - target（只有 present=1 时才有意义）
        var deviation = model.NewIntVar(0, problem.Settings.HorizonMinutes, $"dev_{job.Id}_{machineId}");
        model.Add(deviation >= vars.Start - target).OnlyEnforceIf(vars.Present);
        model.Add(deviation == 0).OnlyEnforceIf(vars.Present.Not());

        deviationTerms.Add(deviation * deviationWeight);
    }
}
```

#### 步骤 10c：目标函数增加偏差项

```csharp
// 修改前：
model.Minimize(LinearExpr.Sum(tardinessTerms)
    + makespan * problem.Settings.MakespanWeight
    + LinearExpr.Sum(transitionTerms));

// 修改后：
model.Minimize(LinearExpr.Sum(tardinessTerms)
    + makespan * problem.Settings.MakespanWeight
    + LinearExpr.Sum(transitionTerms)
    + LinearExpr.Sum(deviationTerms));  // ← 新增目标时间偏差惩罚
```

### 3.5 第五步：`HeuristicScheduler` — 开始时间计算加入目标时间下界

`EvaluateMachine` 和 `BuildMachineSequences` 均需修改：

```csharp
// EvaluateMachine 中：
var start = Math.Max(

        anchor.ReadyMinutes + setup,

```

### 3.6 第六步：新增冲突检测，`SchedulingResponse` 新增 `StartTimeConflicts`

#### `SchedulingResponse` 新增字段

```csharp
// Api/SchedulingResponse.cs
public sealed class SchedulingResponse
{
    // ... 现有字段不变 ...

    /// <summary>开始时间冲突警告列表（仅在 StartTimeStrictness > 0 时可能有内容）。</summary>
    [JsonPropertyName("starttimeconflicts")]
    public IReadOnlyList<StartTimeConflictDto> StartTimeConflicts { get; init; } = [];
}

/// <summary>开始时间冲突警告。</summary>
public sealed class StartTimeConflictDto
{
    /// <summary>工单子件号。</summary>
    [JsonPropertyName("orderchildno")]
    public string OrderChildNo { get; init; } = string.Empty;

    /// <summary>用户指定的目标开始时间。</summary>
    [JsonPropertyName("targetstarttime")]
    public DateTime TargetStartTime { get; init; }

    /// <summary>实际排定的开始时间。</summary>
    [JsonPropertyName("actualstarttime")]
    public DateTime ActualStartTime { get; init; }

    /// <summary>偏移量（分钟），正数表示延迟。</summary>
    [JsonPropertyName("deviationminutes")]
    public int DeviationMinutes { get; init; }

    /// <summary>建议的目标开始时间（= 实际排定时间，即最早可行时间）。</summary>
    [JsonPropertyName("suggestedstarttime")]
    public DateTime SuggestedStartTime { get; init; }

    /// <summary>冲突原因描述。</summary>
    [JsonPropertyName("reason")]
    public string Reason { get; init; } = string.Empty;
}
```

#### `SchedulingEngine.Run` 步骤 7 后追加冲突检测

```csharp
// 步骤 7 之后，构建 response 之前：
var conflicts = BuildStartTimeConflicts(problem, effective.Operations);

var response = new SchedulingResponse
{
    // ... 现有字段 ...
    StartTimeConflicts = conflicts
};
```

```csharp
/// <summary>
/// 检测开始时间冲突：对每个有目标时间的动态工单，比较目标与实际开始时间。
/// </summary>
private static IReadOnlyList<StartTimeConflictDto> BuildStartTimeConflicts(
    SchedulingProblem problem,
    IReadOnlyList<ScheduledOperation> operations)
{
    // 严格度为 0 或无任何工单有目标时间，跳过
    if (problem.Settings.StartTimeStrictness <= 0)
        return [];

    // 只在严格度 >= 30 时才生成警告，避免低严格度下噪音过多
    var warningThreshold = 30;
    if (problem.Settings.StartTimeStrictness < warningThreshold)
        return [];

    var operationLookup = operations
        .Where(x => !x.IsFixed)
        .ToDictionary(x => x.Job.OrderChildNo, StringComparer.OrdinalIgnoreCase);

    var conflicts = new List<StartTimeConflictDto>();

    foreach (var job in problem.Jobs.Where(x => x.TargetStartMinutes.HasValue))
    {
        if (!operationLookup.TryGetValue(job.OrderChildNo, out var op))
            continue;

        var deviation = op.StartMinutes - job.TargetStartMinutes!.Value;
        if (deviation <= 0)
            continue;  // 按时或提前开始，无冲突

        var targetTime = problem.ScheduleTime.AddMinutes(job.TargetStartMinutes.Value);
        var actualTime = problem.ScheduleTime.AddMinutes(op.StartMinutes);

        conflicts.Add(new StartTimeConflictDto
        {
            OrderChildNo = job.OrderChildNo,
            TargetStartTime = targetTime,
            ActualStartTime = actualTime,
            DeviationMinutes = deviation,
            SuggestedStartTime = actualTime,
            Reason = $"机器 {op.MachineId} 在目标时间 {targetTime:HH:mm} 存在资源冲突，" +
                     $"最早可排定时间为 {actualTime:HH:mm}，建议目标开始时间调整为 {actualTime:HH:mm}。"
        });
    }

    return conflicts;
}
```

## 4. 完整数据流

```
宿主 JSON
  └── Settings.StartTimeStrictness = 60（示例）
        │
        ▼
SchedulingPreprocessor.Prepare()
  ├── scheduleTime = DateTime.Now（strictness > 0 时固定取当前时间）
  └── Job.TargetStartMinutes = ceil((PlanStartTime - Now).TotalMinutes)
        │（仅 strictness>0 且 PlanStartTime 在未来时赋值）
        ▼
HeuristicScheduler.Build()
  └── start = max(anchor + setup, moldReady, job.TargetStartMinutes ?? 0)
        │（硬下界：不得早于目标时间）
        ▼
CpSatOptimizer.Optimize()
  ├── 约束：start >= targetStart (OnlyEnforceIf present)
  ├── 偏差变量：deviation = start - targetStart
  └── 目标函数：min(...现有项... + Σ deviation × strictness×4)
        │
        ▼
SchedulingEngine — 步骤 7 后
  └── BuildStartTimeConflicts()
        └── deviation > 0 且 strictness >= 30 → StartTimeConflictDto
              │
              ▼
SchedulingResponse
  ├── ScheduledOrders（含排定时间）
  └── StartTimeConflicts（含冲突工单、目标/实际时间、偏移量、建议时间）
```

## 5. 行为模式说明

### 5.1 strictness = 0（纯优化模式，默认）

```
WO-001: PlanStartTime = 10:00, 时长 60 min
当前时间：09:00, 机器 M1 空闲

排程结果：WO-001 在 09:00 开始（忽略 PlanStartTime）
理由：scheduleTime = 09:00，TargetStartMinutes = null，求解器追求最短总工期
StartTimeConflicts: []
```

**适用场景**：批量全局优化，不关心个别工单的计划时间。

### 5.2 strictness = 50（平衡模式）

```
WO-001: PlanStartTime = 10:00，时长 60 min
当前时间：09:00，机器 M1 空闲

TargetStartMinutes = 60（10:00 - 09:00 = 60 min）
deviationWeight = 50 × 4 = 200

排程结果：WO-001 在 10:00 开始
理由：从 09:00 开始可节省 60 min 工期（makespan 惩罚=60），
      但偏差惩罚 = 60 × 200 = 12000，远大于工期惩罚，
      求解器选择贴近目标时间
StartTimeConflicts: []（无偏差）
```

若机器被另一个工单占用到 10:30：

```
排程结果：WO-001 在 10:30 开始（机器最早可用时间）
偏差 = 30 min，偏差惩罚 = 30 × 200 = 6000
StartTimeConflicts: [{
  OrderChildNo: "WO-001",
  TargetStartTime: "10:00",
  ActualStartTime: "10:30",
  DeviationMinutes: 30,
  SuggestedStartTime: "10:30",
  Reason: "机器 M1 在目标时间 10:00 存在资源冲突，最早可排定时间为 10:30，建议目标开始时间调整为 10:30。"
}]
```

### 5.3 strictness = 100（极高优先模式）

```
deviationWeight = 100 × 4 = 400（= 20 × TardinessWeight）
行为：求解器将开始时间贴近目标视为最高优先级目标
若有冲突：不放弃工单，仍排入最早可行时间，并在 StartTimeConflicts 中记录
```

## 6. 边界情况处理

| 情况 | 处理方式 |
|------|---------|
| `PlanStartTime` 为 null | `TargetStartMinutes = null`，无约束，可立即排 |
| `PlanStartTime` 早于 `DateTime.Now` | `offset <= 0`，`TargetStartMinutes = null`，等同无约束 |
| `PlanStartTime` 超过 `HorizonMinutes` | 标记为不可排产（超出排程窗口） |
| 锚定工单（`shouldAnchor=true`） | 不赋值 `TargetStartMinutes`，时间由 `CreateFixedOperation` 决定 |
| `StartTimeStrictness = 0` | `scheduleTime` 走旧逻辑（最早 PlanStartTime），无目标约束 |
| `StartTimeStrictness < 30` | `TargetStartMinutes` 仍生效（硬下界 + 软惩罚），但不产生冲突警告 |
| 工单本身无 `PlanStartTime` 但同机器有其他工单有目标时间 | 无目标时间的工单不受影响，正常参与优化 |

## 8. 与 v1 方案的对比

| 维度 | v1 方案 | v2 方案 |
|------|--------|--------|
| 模式开关 | `ScheduleTimeMode` 枚举（二选一） | `StartTimeStrictness` 整数滑块（连续渐变） |
| 目标时间语义 | 最早开始下界（不得早于） | 目标时间（尽量在此时间开始，有惩罚） |
| 提前排产 | 允许（只有下界，可以在目标时间开始） | 禁止（硬下界保证不早于目标时间） |
| 资源冲突时 | 自动延期，无警告 | 延期 + 冲突警告 + 建议时间 |
| scheduleTime 取值 | 模式 B 固定为 `DateTime.Now` | strictness>0 时固定为 `DateTime.Now` |
| 单元测试兼容性 | 需更新（新参数 Job/Settings） | 同左，但带默认值，影响最小 |
