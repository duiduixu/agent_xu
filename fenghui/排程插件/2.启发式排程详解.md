- 为什么它是 CP-SAT 的完美"搭档"

## 2. 背景：启发式排程在整体架构中的位置

### 排程流水线全景

```
SchedulingPreprocessor.Prepare()
        │
        │ SchedulingProblem
        ▼
┌─────────────────────────────────────┐
│        SchedulingEngine             │
│                                     │
│  ┌─────────────────────────────┐   │
│  │ ① HeuristicScheduler.Build  │   │  ← ★ 本文档的主角 ★
│  │    生成启发式初排方案          │   │     生成初始可行解
│  │    耗时：< 1 秒              │   │
│  └─────────────┬───────────────┘   │
│                │ ScheduleResult    │
│                │ (Status="Heuristic")│
│                ▼                    │
│  ┌─────────────────────────────┐   │
│  │ ② CpSatOptimizer.Optimize   │   │     精确优化
│  │    以启发式结果为 Hint       │   │     耗时：最多 20 秒
│  │    启动 CP-SAT 搜索          │   │
│  └─────────────┬───────────────┘   │
│                │ ScheduleResult    │
│                │ (Status="Optimal"/"Feasible")│
└────────────────┼───────────────────┘
                 │
                 ▼
           最终排程结果
```

### 为什么启发式排程不是"可有可无"的？

| 场景 | 只用 CP-SAT | 启发式 + CP-SAT |
|------|-----------|-----------------|
| 10 个 Job | ~5 秒找到 Optimal | <1秒(启发式) + ~2秒(CP-SAT) 找到 Optimal |
| 50 个 Job | ~20 秒 Feasible（超时） | <1秒(启发式) + ~20秒(CP-SAT) 找到更好的 Feasible |
| 100+ Job | 20 秒可能无解 | <1秒(启发式)→ 至少有一个可行解 |

**关键优势**：

1. **保证有解**：即使 CP-SAT 在 20 秒内找不到解，还有启发式结果兜底
2. **热启动**：启发式结果作为 CP-SAT 的 Hint，大幅加速搜索
3. **快速响应**：如果情况紧急，可以先用启发式结果应急

## 3. 什么是"启发式"？为什么需要它？

### 启发式算法的定义

**启发式算法**（Heuristic Algorithm）是一种基于经验规则的快速求解方法。它**不保证找到最优解**，但能**在可接受的时间内找到足够好的解**。

**类比**：你在超市找最短结账队伍。精确解法是遍历所有队伍的所有可能排列（NP-hard），你等不了；启发式解法是"找看起来人最少的队伍"——不保证最优，但通常够快也够好。

### 为什么排程问题适合启发式？

排程问题（Job Shop Scheduling）是著名的 **NP-hard 问题**——随着 Job 和机器数量的增长，求解时间呈指数级增长。对于几十个甚至上百个 Job 的问题，精确求解可能在几小时内都无法完成。

启发式算法虽然不保证最优，但它的优势在于：
- **时间复杂度可控**：O(n²) 或 O(n log n)，而不是指数级
- **可解释**：每一步的决策都有清晰的业务逻辑
- **结果可预测**：相同的输入总是产生相同的输出

### 本项目使用的贪心策略

本项目使用**贪心算法**（Greedy Algorithm）——每一步都做"当下看起来最好"的选择，不回退。

```
贪心策略 = 排序 + 逐个分配
  ┌─ 按照某种规则对 Job 排序（优先级排序）
  ├─ 依次取每个 Job，放在"当前最优"的机器上
  └─ 一旦分配，不再改变（不会回退重新分配）
```

## 5. 逐步详解：两阶段贪心策略

### 阶段 0：初始化与验证

```csharp
SchedulingProblemValidator.ValidateProblem(problem);

if (problem.Jobs.Count == 0)
{
    return new ScheduleResult([], 0, "Heuristic");
}

var machineStates = problem.Machines.ToDictionary(
    x => x.Id,
    x => problem.MachineAnchors.TryGetValue(x.Id, out var anchor)
        ? anchor
        : new MachineAnchorState(0, null),
    StringComparer.OrdinalIgnoreCase);

var moldReady = new Dictionary<string, int>(
    problem.MoldReadyTimes, StringComparer.OrdinalIgnoreCase);
```

**这段代码在说什么？**

1. **验证问题**：确保数据结构一致（重复机器、Job 引用的机器不存在等）
2. **空问题快速返回**：没有 Job 就直接返回空结果
3. **复制机器状态**：从 `problem.MachineAnchors` 复制一份机器锚点数据。**为什么要复制？** 因为后续分配过程中会不断更新锚点（每安排一个 Job，机器的锚点就推后了），不能修改原始数据
4. **复制模具就绪时间**：同理，模具就绪时间也会随分配过程而更新

```
初始状态（示例）：
  machineStates = {
    "M1" → { ReadyMinutes: 120, LastJob: Job#001 },
    "M2" → { ReadyMinutes: 0,   LastJob: null },
    "M3" → { ReadyMinutes: 60,  LastJob: Job#003 },
  }
  moldReady = {
    "M001" → 120,   // 模具 M001 最晚 120 分钟就绪
    "M002" → 60,    // 模具 M002 最晚 60 分钟就绪
  }
```

### 阶段 1：AssignJobsToMachines —— 设备分配

这是启发式排程的**第一个核心阶段**——决定每个 Job 去哪台机器。

#### 1a. 初始化分配容器和待分配列表

```csharp
var assignments = problem.Machines.ToDictionary(
    x => x.Id,
    _ => new List<Job>(),
    StringComparer.OrdinalIgnoreCase);

var pending = problem.Jobs
    .OrderBy(x => x.EligibleMachines.Count)       // ① 候选机器少
    .ThenBy(x => x.DueMinutes)                     // ② 交期紧
    .ThenBy(x => x.ColorPriority)                  // ③ 颜色浅
    .ThenBy(x => x.MoldId, StringComparer.OrdinalIgnoreCase)  // ④ 同模具
    .ToList();
```

**为什么这样排序？**

这是一个经典的约束满足问题的贪心策略——**"约束最紧的优先"**（Most Constrained First）。

```
Job 优先级的业务逻辑：

① 候选机器少：只有 1 台机器可选 → 必须优先分配
   如果最后才分配，那台机器可能已经被其他 Job 占满了
   → 选得晚的 Job 应该"生存空间"更宽裕

② 交期紧：DueMinutes 小 → 需要尽快排产
   如果先分配时间充裕的，紧迫的 Job 可能放不下

③ 颜色浅：ColorPriority 小 → 先生产浅色
   注塑行业的"由浅到深"原则——后端 Job 的颜色优先级会递增

④ 同模具：保证同模具的 Job 在列表中是连续的
   后续分配器会倾向于把同模具的 Job 放在同一台机器上
```

#### 1b. 逐个分配：选择最佳机器

```csharp
foreach (var job in pending)
{
    var candidate = job.EligibleMachines
        .Select(machineId => EvaluateMachine(problem, machineStates, moldReady, job, machineId))
        .Where(x => x is not null)
        .OrderByDescending(x => x!.SameMold)        // ① 同模具优先
        .ThenBy(x => x!.Tardiness)                   // ② 延误少
        .ThenBy(x => x!.Setup)                       // ③ 换线时间短
        .ThenBy(x => x!.ColorPenalty)                // ④ 颜色惩罚小
        .ThenBy(x => x!.End)                         // ⑤ 结束早
        .FirstOrDefault();

    if (candidate is null)
    {
        continue; // 所有机器都不兼容此 Job（颜色规则全部禁止）
    }

    assignments[candidate.MachineId].Add(job);
}
```

**决策过程可视化**：

```
Job #XYZ 有待选的机器 [M1, M2, M3]

对每台机器调用 EvaluateMachine：
  M1 → { SameMold=true,  Tardiness=30, Setup=0,  ColorPenalty=0, End=200 }
  M2 → { SameMold=false, Tardiness=10, Setup=30, ColorPenalty=5, End=180 }
  M3 → null（颜色规则不允许）→ 排除

按优先级排序：
  ① SameMold=true  → M1 胜出
  ② (如果 SameMold 相同) Tardiness 小的胜出
  ③ (如果 Tardiness 相同) Setup 小的胜出
  ...

结果：M1 被选中 ✓
```

#### 1c. 更新状态

```csharp
// 注意：在 AssignJobsToMachines 阶段，只记录分配关系，不实际排时间
// machineStates 和 moldReady 在此阶段不被更新
```

**关键理解**：在设备分配阶段，`EvaluateMachine` 只是**模拟**了"如果把 Job 放在这台机器上会怎样"，实际上 `machineStates` 和 `moldReady` 并没有被更新。真正的排时间在阶段 2。

**为什么设备分配阶段不更新状态？**

因为同一个 Job 可能被多个 EvaluateMachine 调用评估，如果每次都更新状态，后续评估就不准确了。而且分配顺序是按 pending 列表走的，不是按机器维度——如果先分配了机器 M1 的 Job A，再分配机器 M2 的 Job B，M1 的实际状态还没更新，M2 的评估就会"乐观"（因为不知道 M1 会被占用）。

这是一个简化设计——不完美，但足够快。CP-SAT 会在后续弥补这个简化带来的不精确。

### 阶段 2：BuildMachineSequences —— 设备内排序

这是启发式排程的**第二个核心阶段**——在每台机器上，决定 Job 的先后顺序。

#### 2a. 初始化

```csharp
var scheduled = new List<ScheduledOperation>();
var moldReady = new Dictionary<string, int>(
    problem.MoldReadyTimes, StringComparer.OrdinalIgnoreCase);
```

重新初始化 `moldReady`（因为阶段 1 没有修改它），为每台机器独立排时间。

#### 2b. 逐台机器排程

```csharp
foreach (var machine in problem.Machines.OrderBy(x => x.Id))
{
    if (!assignments.TryGetValue(machine.Id, out var assignedJobs)
        || assignedJobs.Count == 0)
    {
        continue;
    }

    // 获取机器锚点
    var anchor = problem.MachineAnchors.TryGetValue(machine.Id, out var machineAnchor)
        ? machineAnchor
        : new MachineAnchorState(0, null);

    // 初始排序：交期 → 颜色 → 模具
    var remaining = assignedJobs
        .OrderBy(x => x.DueMinutes)
        .ThenBy(x => x.ColorPriority)
        .ThenBy(x => x.MoldId, StringComparer.OrdinalIgnoreCase)
        .ToList();

    // 逐个选择下一个 Job
    while (remaining.Count > 0)
    {
        var next = SelectNextJob(problem, anchor.LastJob, remaining);
        if (next is null)
        {
            break; // 没有兼容的 Job 了（颜色规则全部禁止）
        }

        // 计算实际开始时间和结束时间
        var setup = _setupCalculator.Compute(anchor.LastJob, next, problem.ScheduleConfig);
        var start = Math.Max(
            anchor.ReadyMinutes + setup,        // 机器就绪 + 换线时间
            moldReady.TryGetValue(next.MoldId, out var mrt) ? mrt : 0); // 模具就绪
        var end = start + next.DurationMinutes;

        var operation = new ScheduledOperation(
            next, machine.Id, start, end,
            Math.Max(0, end - next.DueMinutes));

        scheduled.Add(operation);
        anchor = new MachineAnchorState(end, next);    // 更新锚点
        moldReady[next.MoldId] = end;                   // 更新模具就绪
        remaining.Remove(next);                         // 从待排列表中移除
    }
}
```

**算法解读**：这是一个"循环贪心"算法。

```
┌─────────────────────────────────────────────┐
│  while (remaining.Count > 0)                 │
│  {                                           │
│    ① 从 remaining 中选出最好的 next Job      │
│       （SelectNextJob — 基于贪心规则）        │
│                                              │
│    ② 把 next 安排到时间轴上                  │
│       start = max(机器就绪, 模具就绪)         │
│       end = start + duration                 │
│                                              │
│    ③ 更新状态：                              │
│       - anchor = (end, next)  // 机器锚点推后│
│       - moldReady[moldId] = end // 模具占用   │
│       - remaining.Remove(next)               │
│                                              │
│    ④ 回到 ①，直到 remaining 为空              │
│  }                                           │
└─────────────────────────────────────────────┘
```

**可视化示例**：机器 M1 上有 3 个 Job（A、B、C）

```
初始状态：
  机器 M1: 锚点=60（固定工序在 60 分钟结束）
  moldReady = { M001: 30, M002: 45, M003: 0 }

第 1 轮 SelectNextJob：
  remaining = [A(M001), B(M002), C(M003)]
  前一个 Job = anchor.LastJob（固定工序的 Job）
  → 选中 B（因为 B 的规则最匹配）
  → setup = 30（从 anchor.LastJob 切换到 B 需要 30 分钟）
  → start = max(60+30, 45) = 90
  → end = 90 + 20(生产时长) = 110

  时间轴：M1 [60固定]～～setup30～～[B 90→110]

第 2 轮 SelectNextJob：
  remaining = [A(M001), C(M003)]
  前一个 Job = B
  → 选中 A（因为 A 和 B 同模具或颜色兼容）
  → setup = 0（同模具无需换线）
  → moldReady["M001"] = 30（不变，因为 A 在 M001 上排得晚）
  → start = max(110+0, 30) = 110
  → end = 110 + 15 = 125

  时间轴：M1 [60固定]～～setup30～～[B 90→110][A 110→125]

第 3 轮 SelectNextJob：
  remaining = [C(M003)]
  前一个 Job = A
  → 选中 C
  → setup = 30（换模具）
  ... 依次类推
```

### 阶段 3：计算目标值并返回

```csharp
var objective = _objectiveCalculator.Calculate(
    problem.FixedOperations.Concat(scheduled)
        .OrderBy(x => x.StartMinutes).ToList(),
    problem.Settings,
    problem.ScheduleConfig,
    problem.ColorPolicy);

return new ScheduleResult(scheduled, objective, "Heuristic");
```

**ObjectiveCalculator 做了什么？**

将固定工序和动态工序合并后，计算四项加权和：

```
目标值 = 延误总分钟 × TardinessWeight(20)
       + 最晚结束时间 × MakespanWeight(1)
       + 换线总时间 × SetupWeight(1)
       + 颜色总惩罚 × ColorBacktrackWeight(180)
```

**返回值**：

```csharp
new ScheduleResult(
    scheduled,          // 排好的工序列表（仅动态 Job，不含固定工序）
    objective,          // 目标函数值
    "Heuristic")        // 状态标记 = 启发式结果
```

这个目标值可以和 CP-SAT 的结果进行比较，评估 CP-SAT 的优化效果。

## 6. 关键子方法详解

### 6.1 EvaluateMachine：机器评估（设备分配阶段的核心）

```csharp
private MachineCandidate? EvaluateMachine(
    SchedulingProblem problem,
    IReadOnlyDictionary<string, MachineAnchorState> machineStates,
    IReadOnlyDictionary<string, int> moldReady,
    Job job,
    string machineId)
{
    var anchor = machineStates[machineId];

    // 检查 1：颜色规则是否允许
    if (!problem.ColorPolicy.IsAllowed(anchor.LastJob, job))
    {
        return null;  // 不兼容，排除此机器
    }

    // 计算换线时间
    var setup = _setupCalculator.Compute(anchor.LastJob, job, problem.ScheduleConfig);

    // 计算开始时间
    var start = Math.Max(
        anchor.ReadyMinutes + setup,
        moldReady.TryGetValue(job.MoldId, out var moldReadyTime) ? moldReadyTime : 0);

    var end = start + job.DurationMinutes;
    var tardiness = Math.Max(0, end - job.DueMinutes);

    return new MachineCandidate(
        machineId,
        start,                                      // Start — 开始时间
        end,                                        // End — 结束时间
        tardiness,                                  // Tardiness — 延误分钟
        setup,                                      // Setup — 换线时间
        problem.ColorPolicy.GetPenalty(anchor.LastJob, job),  // ColorPenalty
        anchor.LastJob is not null
            && string.Equals(anchor.LastJob.MoldId, job.MoldId,
                StringComparison.OrdinalIgnoreCase));  // SameMold
}
```

**评估逻辑流程图**：

```
对机器 machineId，评估 Job job：

  ① 获取该机器的当前锚点状态
  ② 检查颜色规则：
     该机器上一个 Job → 当前 Job 的颜色切换是否允许？
       ├── 不允许 → return null（此机器不可选）
       └── 允许 → 继续
  ③ 计算换线时间（setup）：
      上一个 Job → 当前 Job 是否需要换模？
  ④ 计算最早开始时间：
      start = max(机器就绪+换线, 模具就绪)
  ⑤ 计算结束时间：
      end = start + 生产时长
  ⑥ 计算延误：
      tardiness = max(0, end - 交期)
  ⑦ 返回评估结果（MachineCandidate）
```

**为什么返回 null 而不继续评估？**

如果颜色规则不允许，这台机器就完全不可用。没有继续评估的意义——不管延误多少，颜色的硬约束必须先满足。

### 6.2 SelectNextJob：下一个 Job 选择（设备内排序阶段的核心）

```csharp
private static Job? SelectNextJob(
    SchedulingProblem problem,
    Job? previous,
    IReadOnlyList<Job> remaining)
{
    return remaining
        .Where(x => problem.ColorPolicy.IsAllowed(previous, x))       // 过滤
        .OrderByDescending(x => previous is not null
            && string.Equals(previous.MoldId, x.MoldId,
                StringComparison.OrdinalIgnoreCase))                   // ① 同模具
        .ThenByDescending(x => previous is not null
            && string.Equals(previous.ColorName, x.ColorName,
                StringComparison.OrdinalIgnoreCase))                   // ② 同颜色
        .ThenBy(x => previous is null ? 0 :
            Math.Abs(previous.ColorPriority - x.ColorPriority))        // ③ 近色
        .ThenBy(x => x.ColorPriority)                                  // ④ 浅色→深色
        .ThenBy(x => x.DueMinutes)                                     // ⑤ 交期紧
        .ThenBy(x => x.MoldId, StringComparer.OrdinalIgnoreCase)       // ⑥ 稳定排序
        .FirstOrDefault();
}
```

**选择逻辑逐级拆解**：

```
从 remaining 中选择"最好"的下一个 Job

第 0 步：过滤
  排除颜色规则不允许的 Job（IsAllowed 返回 false）
  ↓ 过滤后的列表

第 1 步排序（优先级从高到低）：

  ① 同模具优先（OrderByDescending）
     → previous.Job 和当前 Job 使用同一个模具？
     → 是 → 排在前面（换线成本为 0，连续生产）
     → 否 → 排在后面

  ② 同颜色优先（ThenByDescending）
     → previous.Job 和当前 Job 颜色相同？
     → 是 → 排在前面（不需要清洗设备）
     → 否 → 排在后面

  ③ 近色优先（ThenBy，升序）
     → |previous.Priority - current.Priority| 越小越好
     → 颜色越接近，切换成本越低

  ④ 浅色→深色（ThenBy，升序）
     → ColorPriority 小的先生产（浅色先生产）
     → 遵循"由浅到深"的行业规则

  ⑤ 交期紧优先（ThenBy，升序）
     → DueMinutes 小的先生产
     → 降低延误风险

  ⑥ 同模具稳定排序（ThenBy，升序）
     → 相同条件下，按 MoldId 排序，保证确定性
```

**可视化示例**：

```
previous.Job = { MoldId: "M001", ColorName: "白色", ColorPriority: 10 }

remaining = [
  A: { MoldId: "M001", Color: "白色",  Priority: 10, Due: 200 }  ← 最匹配!
  B: { MoldId: "M002", Color: "米白",  Priority: 12, Due: 150 }
  C: { MoldId: "M001", Color: "浅灰",  Priority: 20, Due: 300 }
  D: { MoldId: "M003", Color: "黑色",  Priority: 50, Due: 100 }
  E: { MoldId: "M002", Color: "深灰",  Priority: 40, Due:  80 }  ← 颜色规则可能禁止
]

排序结果：
  ① 同模具：A✓ C✓ 排前面
  ② 同颜色：在①基础上再分
  ③ 近色：颜色接近 previous 的优先
  ④ 浅→深：浅色优先
  ⑤ 交期：交期紧的优先

最终排序：A > C > B > D > E（或 E 被 IsAllowed 过滤掉）

→ 选择 A 作为下一个 Job
```

### 6.3 MachineCandidate：评估结果数据结构

```csharp
private sealed record MachineCandidate(
    string MachineId,    // 机器 ID
    int Start,           // 预计开始分钟数
    int End,             // 预计结束分钟数
    int Tardiness,       // 预计延误分钟数 (max(0, End - Due))
    int Setup,           // 换线/换模时间（分钟）
    int ColorPenalty,    // 颜色切换惩罚分数（0/1/3/8/15）
    bool SameMold);      // 是否与上一个 Job 使用同一模具
```

**这是一个纯数据容器（record）**，不需要额外解释。它在 `EvaluateMachine` 中创建，被外层用于排序和选择。

## 7. 贪心排序策略详解

这个启发式排程器有三个关键的排序位置，每个都有不同的业务含义。

### 7.1 设备分配时的 Job 排序

在 `AssignJobsToMachines` 中，待分配的 Job 列表按以下顺序排列：

```
① EligibleMachines.Count  ↑（候选机器少 → 约束紧 → 先分配）
② DueMinutes              ↑（交期紧 → 时间紧迫 → 先分配）
③ ColorPriority           ↑（颜色优先级小 → 浅色 → 先分配）
④ MoldId                  ↑（稳定排序）
```

**策略名称**："最受约束者优先"（Most Constrained First）

| 为什么要这样排 | 原因 |
|---------------|------|
| 候选机器少先排 | 只有 1 台可选 → 必须确保那台有空位。如果放到最后，可能被占满 |
| 交期紧先排 | 越紧迫的 Job 越需要尽早获得"好位置" |
| 浅色先排 | 只有浅色先排，后续深色才能"接在后面"，整体才符合"浅→深" |
| 同模具 | 使同模具 Job 在列表中连续，分配器可能选同一台机器 |

### 7.2 设备分配时的机器选择排序

在 `AssignJobsToMachines` 中，对同一 Job 的各台机器评估结果进行排序：

```
① SameMold     ↓（同模具 → 排最前面 → 换线成本为 0）
② Tardiness    ↑（延误少 → 交期目标）
③ Setup        ↑（换线时间短 → 换线成本低）
④ ColorPenalty ↑（颜色惩罚小 → 颜色兼容好）
⑤ End          ↑（结束早 → 不浪费机器时间）
```

**策略名称**："多目标加权排序"（Multi-Objective Lexicographic Ordering）

注意：这里使用的是**字典序**（Lexicographic）而非加权求和。即先比 SameMold，SameMold 相同再比 Tardiness，依此类推。这避免了不同量纲指标之间的权重调优问题。

### 7.3 设备内排序时的 Job 选择排序

在 `BuildMachineSequences` 中，用 `SelectNextJob` 选择下一个要排的 Job：

```
① 颜色规则过滤（IsAllowed）  ← 硬约束：不过滤的不能排
② 同模具 ↓（换线时间=0，最优选择）
③ 同颜色 ↓（不需要深度清洗）
④ 近色   ↑（颜色差异小）
⑤ 浅→深  ↑（遵循行业规则）
⑥ 交期紧 ↑（降低延误）
⑦ 同模具 ↑（稳定排序）
```

**策略名称**："颜色友好优先"（Color-Friendly First）

这个排序最注重颜色连续性——同模具、同颜色、近色的 Job 被排在一起，可以大幅减少换模次数和清洗成本。

## 8. 与 CP-SAT 的对比

| 维度 | 启发式排程器 | CP-SAT 优化器 |
|------|------------|-------------|
| **算法类型** | 贪心（构造性） | 精确 + 启发式搜索（CP-SAT） |
| **时间复杂度** | O(n²) | 指数级（20 秒截断） |
| **解的质量** | 可行解，不保证最优 | 接近最优（小问题）或 改进解（大问题） |
| **是否保证找到解** | 几乎总是能找到 | 20 秒内可能找不到可行解 |
| **是否可解释** | 每一步都清晰可理解 | 黑盒——只知道最终结果 |
| **结果确定性** | 完全确定性（相同输入→相同输出） | 可能不同（多线程搜索的不确定性） |
| **对 Hint 的需求** | 不需要 | 有 Hint 大幅加速 |
| **典型耗时** | < 100ms（100 个 Job） | 0.1~20 秒 |

**两者配合**：

```
启发式结果 → CP-SAT 的 Hint
  ├── CP-SAT 在启发式解附近搜索更优解
  ├── 如果 20 秒内找到更优解 → 用 CP-SAT 结果
  └── 如果 20 秒后不如启发式 → 回退到启发式结果
```

**启发式"回退"的典型情况**：

- Job 数量极少（< 5 个）：启发式结果通常就是最优解
- 权重配置极端：贪心策略不能很好反映权重的优先级
- 颜色约束极严：贪心的"最近看"可能不如"全局看"

在本项目的 `SchedulingEngine` 中，CP-SAT 结果如果比启发式差，会回退到启发式并标记 `Status = "HeuristicFallback"`。

## 9. 常见疑问解答（FAQ）

### Q1：为什么设备分配阶段不实际排时间？

因为在设备分配阶段，我们还没有决定同机器上 Job 的顺序。如果提前排时间，后续 Job 会被迫排在更晚的时间——而在 `BuildMachineSequences` 阶段，可能发现更好的顺序（比如同模具的 Job 放在一起），实际排出来的时间会比预估值更紧凑。

**EvaluateMachine 只是一个"探头"**——它说："如果能排在这台机器上，大概会是这个效果"，实际排的时候可能更好。

### Q2：SelectNextJob 返回 null 怎么办？

如果 `remaining` 中所有 Job 都被颜色规则禁止（`IsAllowed` 返回 false），`SelectNextJob` 会返回 null。此时 `while` 循环会 `break` 退出——剩余的 Job 不会被排程。

**这正常吗？** 在正常的数据配置下不应该发生（因为颜色规则不应该让所有剩余 Job 都不兼容）。但如果发生了，说明颜色规则配置有问题（比如把所有颜色切换都禁止了）。

### Q3：启发式结果的目标值和 CP-SAT 的目标值可以直接比较吗？

可以。两者使用相同的 `ObjectiveCalculator.Calculate` 计算目标值，公式完全一致：

```
目标值 = 延误×20 + makespan×1 + 换线×1 + 颜色惩罚×180
```

越小越好。如果 CP-SAT 的目标值 ≤ 启发式的目标值，CP-SAT 的结果更好（或至少不差）。

### Q4：为什么 AssignJobsToMachines 中的排序和 SelectNextJob 中的排序不一样？

因为它们解决的是不同的问题：

- **AssignJobsToMachines**：给定一个 Job，选**机器**——关注机器的兼容性和"空闲度"
- **SelectNextJob**：给定一个机器，选**Job**——关注 Job 之间的颜色连续性和交期

这是同一个问题的两个对称视角，排序规则自然不同。

### Q5：buildMachineSequences 中每条机器独立排程，会不会互相冲突？

会的——它们通过 `moldReady` 共享模具就绪时间。如果机器 M1 排了使用模具 M001 的 Job，`moldReady["M001"]` 会被更新为结束时间，机器 M2 排另一个使用 M001 的 Job 时必须等到 M1 的 Job 结束之后。

```
M1: [Job A (M001) 60→90]  → moldReady["M001"] = 90
M2: [Job B (M001)]         → start ≥ 90（必须等 M1 的 A 结束）
    [Job B (M001) 90→120]  → moldReady["M001"] = 120
```

### Q6：这个启发式算法有名字吗？

它属于**列表调度算法**（List Scheduling）家族，具体是"两阶段贪心构造算法"：

- 阶段 1：基于规则的分配（Rule-Based Assignment）
- 阶段 2：基于优先级的序列构造（Priority-Based Sequence Construction）

在排程文献中，类似的算法被称为**分派规则**（Dispatching Rules），如 EDD（Earliest Due Date）、SPT（Shortest Processing Time）等的组合。
