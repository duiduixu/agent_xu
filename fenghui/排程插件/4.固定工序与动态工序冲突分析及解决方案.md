# 固定工序与动态工序时间冲突分析及解决方案

## 1. 问题背景与核心疑问

用户的疑问是：**在排程结果最后合并固定工序与动态工序时，是否会出现同机器、同模具的时间段互相重叠，导致排产结果不可信？**

要回答这个问题，需要彻底搞清楚：

1. "排程基准时间 `scheduleTime`"在整个流程中究竟起什么作用
2. 固定工序是什么时候被创建的，它的时间约束是怎么传递到求解器的
3. 求解器是否真的知道固定工序的存在，并在约束中排除了它们

---

## 2. 现有架构的完整数据流

```
SchedulingInput
  └── WorkOrders（原始工单列表）
        │
        ▼
SchedulingPreprocessor.Prepare()
  ├── 计算 scheduleTime（全局基准时间坐标原点）
  ├── 遍历工单分类：
  │     ├── shouldAnchor = true  → CreateFixedOperation()
  │     │     → ScheduledOperation（IsFixed=true，StartMinutes/EndMinutes 相对于 scheduleTime）
  │     └── shouldOptimize = true → dynamicJobs（待求解器分配的 Job）
  │
  ├── 构建 machineAnchors：每台机器上最后一个固定工序的 EndMinutes + Job
  │     → 传入 SchedulingProblem
  │
  └── 构建 moldReadyTimes：每套模具上所有固定工序的最大 EndMinutes
        → 传入 SchedulingProblem
             │
             ▼
HeuristicScheduler.Build() / CpSatOptimizer.Optimize()
  └── 使用 machineAnchors 和 moldReadyTimes 约束动态 Job 的最早开始时间：
        start ≥ anchor.ReadyMinutes + setupTime
        start ≥ moldReadyTimes[moldId]
             │
             ▼
SchedulingEngine.Run() 步骤 5
  └── allOperations = FixedOperations.Concat(effective.Operations)
        → 最终合并输出
```

## 3. 逐项冲突可能性分析

### 3.1 同一台机器上，固定工序与动态工序时间重叠？

**结论：设计上不会，但存在一个真实的边界漏洞。**

**为什么设计上不会：**

预处理器在阶段 4 构建了 `machineAnchors`：

```csharp
// SchedulingPreprocessor.cs 阶段 4
var machineAnchors = orderedFixedOperations
    .GroupBy(x => x.MachineId)
    .ToDictionary(x => x.Key, x => {
        var tail = x.OrderByDescending(y => y.EndMinutes).First();
        return new MachineAnchorState(tail.EndMinutes, tail.Job);
    });
```

锚点的 `ReadyMinutes` 就是该机器上固定工序的最晚结束时间。启发式和 CP-SAT 都用它约束动态 Job 的开始时间：

```csharp
// HeuristicScheduler — EvaluateMachine
var start = Math.Max(anchor.ReadyMinutes + setup, moldReady[job.MoldId]);

// CpSatOptimizer — 步骤 4b
var minStart = Math.Max(anchor.ReadyMinutes + initialSetup, moldReady);
model.Add(start >= minStart).OnlyEnforceIf(present);
```

因此，**同机器上的动态工序在时间上必然在固定工序之后**——这个保证是通过约束传递实现的，不依赖最后合并时的排序。

**真实的边界漏洞：**

`machineAnchors` 只记录了每台机器**最后一个**固定工序的结束时间（取 `EndMinutes` 最大值）。但如果一台机器有多个固定工序，且它们之间存在时间间隙，动态工序仍然只能排在最后一个固定工序之后，**无法插入间隙**。

```
示例：机器 M1 有两个固定工序
  固定工序 A：[0 ─────── 60]
  固定工序 B：[90 ─────── 150]  ← 最晚，anchor.ReadyMinutes = 150

  间隙：[60, 90] 共 30 分钟，动态工序不会被排进来
  动态工序 C（时长 20 分钟）必须从 150 之后开始

  时间轴：[A: 0→60]  [gap 30]  [B: 90→150]  [C: 150→170]

这是正确的。C 不会"插入"到 A 和 B 之间——这避免了冲突。
```

**这个行为是正确的**：间隙不利用是保守策略，但不会产生冲突。

---

### 3.2 同一套模具，固定工序与动态工序时间重叠？

**结论：设计上不会。**

预处理器在阶段 5 构建了 `moldReadyTimes`：

```csharp
var moldReadyTimes = orderedFixedOperations
    .GroupBy(x => x.Job.MoldId)
    .ToDictionary(x => x.Key, x => x.Max(y => y.EndMinutes));
```

这是该模具上**所有**固定工序的最大结束时间。动态 Job 使用该模具时，开始时间必须 ≥ `moldReadyTimes[moldId]`，且 CP-SAT 中还额外加了 `AddNoOverlap` 约束覆盖动态 Job 之间的模具冲突。

固定工序本身不创建 CP-SAT 的 `IntervalVar`，所以不参与 `AddNoOverlap`——但 `moldReadyTimes` 作为下界约束等效实现了"固定工序占用模具期间，动态工序不能开始"的语义。

---

### 3.3 真正的问题所在：scheduleTime 的取值导致固定工序时间计算偏差

**这才是当前架构中最真实、最严重的潜在问题。**

#### 问题一：scheduleTime 取"最早 PlanStartTime"，导致固定工序的相对时间出现负数或不准确

当前代码：

```csharp
// SchedulingPreprocessor.cs 第 77~80 行
var scheduleTime = input.WorkOrders
    .Where(x => x.Status == WorkOrderStatus.PendingSchedule && x.PlanStartTime != null)
    .OrderBy(x => x.PlanStartTime)
    .FirstOrDefault()?.PlanStartTime ?? DateTime.Now;
```

**场景复现：**

```
当前真实时间：2025-01-15 09:00

工单列表：
  WO-001 (待排产):  PlanStartTime = 2025-01-15 08:00  ← 最早
  WO-002 (生产中):  ActualStartTime = 2025-01-15 07:30, DeviceId = M1  ← shouldAnchor=true
  WO-003 (待排产):  PlanStartTime = 2025-01-15 10:00

scheduleTime = 2025-01-15 08:00（取最早 PlanStartTime）

CreateFixedOperation 处理 WO-002：
  startTime = ActualStartTime = 2025-01-15 07:30
    但 ActualStartTime(07:30) < scheduleTime(08:00)
    → 代码修正：startTime = scheduleTime = 08:00  ✓（这里有保护逻辑）

  endTime = PlanEndTime ?? startTime + DurationMinutes
    如果 PlanEndTime = 2025-01-15 11:00
    startMinutes = (08:00 - 08:00) = 0
    endMinutes = (11:00 - 08:00) = 180

MachineAnchor[M1].ReadyMinutes = 180

→ 动态工序在 M1 上最早从 180 分钟（即 11:00）开始  ✓ 正确
```

这个场景是**安全的**，因为 `CreateFixedOperation` 内部有 `ActualStartTime <= scheduleTime` 的修正逻辑。

#### 问题二：scheduleTime 早于 DateTime.Now 时，固定工序的时间轴与"当前时间"的语义错位

```
当前真实时间：2025-01-15 09:00

WO-001 (待排产):  PlanStartTime = 2025-01-15 07:00  ← scheduleTime = 07:00
WO-002 (调机中):  PlanStartTime = 2025-01-15 08:00, PlanEndTime = 2025-01-15 10:00, DeviceId = M1

CreateFixedOperation(WO-002):
  startTime = PlanStartTime = 08:00 > scheduleTime(07:00) → 无修正
  startMinutes = (08:00 - 07:00) = 60
  endMinutes = (10:00 - 07:00) = 180

MachineAnchor[M1].ReadyMinutes = 180

排程结果输出：
  ApsStartTime = scheduleTime.AddMinutes(60) = 07:00 + 60min = 08:00  ✓
  动态工序从 180 分钟（10:00）之后开始  ✓
```

这个场景也是**安全的**，因为固定工序的绝对时间来源于工单的 `PlanStartTime`/`PlanEndTime`，转换回绝对时间时结果正确。

#### 问题三：scheduleTime 与 WO-001（待排产工单）的 PlanStartTime 早于当前时间

这才是真正可能导致"排产时间不准"的场景：

```
当前真实时间：2025-01-15 09:00

WO-001 (待排产):  PlanStartTime = 2025-01-15 07:00  ← scheduleTime = 07:00

动态排程从 scheduleTime=07:00 开始排。
结果：某个动态工序 StartMinutes=0，
ApsStartTime = 07:00 + 0min = 07:00

但现在已经是 09:00 了！排程输出的开始时间 07:00 已经在过去。
宿主系统看到 ApsStartTime=07:00，而当前时间是 09:00——这个工单"应该已经开始了"。
```

**这是真实的语义问题**：排产结果中动态工序的 `ApsStartTime` 可能落在"过去"，导致宿主系统误判工单状态。

---

### 3.4 同机器上固定工序之间是否会互相重叠？

**结论：不在排程器的职责范围内，但可能发生。**

固定工序的时间来自宿主系统传入的 `PlanStartTime`/`PlanEndTime`/`ActualStartTime`。如果宿主系统传入的数据本身有重叠（比如同一台机器上两个"生产中"的工单时间段重叠），排程器**不检测也不处理**，只是原样放入 `fixedOperations`，并取 `EndMinutes` 最大值作为锚点。

这不会导致排程引擎崩溃，但最终输出的 `allOperations` 中固定工序之间会有重叠——这是输入数据的问题，应该在预处理阶段的验证中捕获。

---

### 3.5 动态工序之间会相互冲突吗？

**结论：不会（有 CP-SAT 的 AddNoOverlap 保证）。**

CP-SAT 的步骤 7 和步骤 8 分别添加了机器互斥和模具互斥约束：

```csharp
// 步骤 7：机器互斥
model.AddNoOverlap(intervals);  // 同机器上的区间不重叠

// 步骤 8：模具互斥
model.AddNoOverlap(intervals);  // 同模具的区间不重叠
```

这两个约束在数学上保证了动态工序之间的合法性。

## 4. 问题总结

| 编号 | 问题 | 严重程度 | 实际是否存在 |
|------|------|---------|-------------|
| P1 | 动态工序与固定工序在**同机器**时间重叠 | 高 | **不存在**（MachineAnchor 约束保护） |
| P2 | 动态工序与固定工序在**同模具**时间重叠 | 高 | **不存在**（MoldReadyTimes 约束保护） |
| P3 | 动态工序之间时间重叠 | 高 | **不存在**（CP-SAT AddNoOverlap 保护） |
| P4 | `scheduleTime` 取过去时间，导致动态工序的 `ApsStartTime` 落在过去 | 中 | **存在**（PlanStartTime 早于 DateTime.Now 时触发） |
| P5 | 固定工序之间互相重叠（宿主数据问题） | 中 | **存在**（宿主传入脏数据时，排程器不检测） |
| P6 | 固定工序之间的间隙未被动态工序利用 | 低 | **存在**（保守策略，不是 bug，是限制） |
| P7 | scheduleTime 的 OrderByDescending 写法误取"最晚"而非"最早" | 中 | **存在**（代码 bug，见下文） |

## 5. 发现的 Bug：scheduleTime 取值逻辑错误

### 5.1 定位

```csharp
// SchedulingPreprocessor.cs 第 77~80 行（当前代码）
var scheduleTime = input.WorkOrders
    .Where(x => x.Status == WorkOrderStatus.PendingSchedule && x.PlanStartTime != null)
    .OrderByDescending(x => x.PlanStartTime)   // ← 注意：OrderByDescending = 降序 = 最晚
    .FirstOrDefault(x => x.Status == WorkOrderStatus.PendingSchedule)?.PlanStartTime
    ?? DateTime.Now;
```

**`.OrderByDescending().FirstOrDefault()` 取的是最晚的 PlanStartTime，不是最早的！**

注释写的是"取最早"，但实际代码取的是"最晚"——这是一个**注释与实现不一致的 bug**。

> 注：在之前生成的《预处理排程详解教程》文档中已提到此处逻辑，当时的代码样本取的是"最晚"（OrderByDescending），和注释"取最早"矛盾。此 bug 在本文分析过程中得到了确认。

### 5.2 影响

如果待排产工单的 PlanStartTime 分散（比如有今天和明天的），取"最晚"意味着 `scheduleTime` 被推到明天，今天的工单的 `StartMinutes` 可能变成负数（虽然 `Math.Max(0,...)` 会修正，但 `DueMinutes` 的计算也会受影响）。

---

## 6. 解决方案

### 方案 A：修复 scheduleTime 取值 Bug（必做）

将 `OrderByDescending` 改为 `OrderBy`，确保取的是"最早"的 PlanStartTime：

```csharp
// 修复后：
var scheduleTime = input.WorkOrders
    .Where(x => x.Status == WorkOrderStatus.PendingSchedule && x.PlanStartTime != null)
    .OrderBy(x => x.PlanStartTime)                    // ← 升序，取最早
    .FirstOrDefault()?.PlanStartTime
    ?? DateTime.Now;
```

### 方案 B：scheduleTime 不早于 DateTime.Now（必做）

当 PlanStartTime 早于当前时间时，`scheduleTime` 应取 `DateTime.Now`，避免动态工序的 `ApsStartTime` 落在过去：

```csharp
var earliestPlanStart = input.WorkOrders
    .Where(x => x.Status == WorkOrderStatus.PendingSchedule && x.PlanStartTime != null)
    .OrderBy(x => x.PlanStartTime)
    .FirstOrDefault()?.PlanStartTime;

var scheduleTime = earliestPlanStart.HasValue && earliestPlanStart.Value > DateTime.Now
    ? earliestPlanStart.Value   // 计划开始在未来，以它为基准
    : DateTime.Now;             // 否则以当前时间为基准（含"无计划时间"的情况）
```

**语义**：`scheduleTime` 表示"排程可以从这个时间点开始安排工单"。它不应该早于"现在"，否则排出来的时间计划已经过期。

### 方案 C：固定工序数据有效性校验（建议做）

在预处理器阶段 4 之后，增加对固定工序的有效性检查，发现同机器时间重叠时打印警告日志（不抛异常，避免影响正常排程流程）：

```csharp
// 在阶段 4 末尾（machineAnchors 构建完毕后）添加：
foreach (var group in orderedFixedOperations.GroupBy(x => x.MachineId, StringComparer.OrdinalIgnoreCase))
{
    ScheduledOperation? prev = null;
    foreach (var op in group.OrderBy(x => x.StartMinutes))
    {
        if (prev is not null && op.StartMinutes < prev.EndMinutes)
        {
            // 发现重叠，记录警告（不影响后续排程，锚点取最大 EndMinutes 已自动处理）
            _logger?.LogWarning(
                "机器 {MachineId} 上固定工序存在时间重叠：{OrderA}({AStart}~{AEnd}) 与 {OrderB}({BStart}~{BEnd})",
                group.Key,
                prev.Job.OrderChildNo, prev.StartMinutes, prev.EndMinutes,
                op.Job.OrderChildNo, op.StartMinutes, op.EndMinutes);
        }
        prev = op;
    }
}
```

> **注意**：`SchedulingPreprocessor` 当前是无状态的纯逻辑类，没有 `ILogger` 注入。若要加日志，需通过构造函数注入 `ILogger<SchedulingPreprocessor>`，或在 `SchedulingEngine.Run()` 的步骤 1 完成后由 Engine 做这个校验。推荐后者，避免改变 Preprocessor 的签名。

### 方案 D：scheduleTime 精确到分钟（建议做）

当前 `DateTime.Now` 精确到毫秒，但所有时间计算都用 `(int)Math.Ceiling(...)` 向上取整到分钟。将 `scheduleTime` 在赋值时截断到分钟，可以消除小数点秒级误差导致的奇怪偏移：

```csharp
// 将 scheduleTime 截断到分钟（去掉秒和毫秒）
var scheduleTime = ...;   // 现有逻辑
scheduleTime = new DateTime(
    scheduleTime.Year, scheduleTime.Month, scheduleTime.Day,
    scheduleTime.Hour, scheduleTime.Minute, 0, scheduleTime.Kind);
```

## 8. 修复后的完整代码片段（`SchedulingPreprocessor.cs` 阶段 2 开头）

```csharp
// ===================================================================
// 阶段 2：构建映射表
// ===================================================================

// 排程基准时间：
// - 取"待排产"工单中最早的 PlanStartTime
// - 若该时间早于或等于当前时间（或无有效 PlanStartTime），则以 DateTime.Now 为基准
// - 时间精确到分钟，消除秒级偏差
var earliestPlanStart = input.WorkOrders
    .Where(x => x.Status == WorkOrderStatus.PendingSchedule && x.PlanStartTime != null)
    .OrderBy(x => x.PlanStartTime)      // 升序，取最早
    .FirstOrDefault()?.PlanStartTime;

var rawScheduleTime = earliestPlanStart.HasValue && earliestPlanStart.Value > DateTime.Now
    ? earliestPlanStart.Value
    : DateTime.Now;

// 截断到分钟，避免秒级舍入带来的偏差
var scheduleTime = new DateTime(
    rawScheduleTime.Year, rawScheduleTime.Month, rawScheduleTime.Day,
    rawScheduleTime.Hour, rawScheduleTime.Minute, 0,
    rawScheduleTime.Kind == DateTimeKind.Unspecified ? DateTimeKind.Local : rawScheduleTime.Kind);
```

## 9. 结论

**直接回答原始问题：**

> 在 CP-SAT 优化后合并固定工序与动态工序，会不会出现时间、机器、模具冲突？

**不会发生**（在正常数据输入下）。现有架构通过以下三重保护确保了合并结果的合法性：

1. **MachineAnchor**：确保动态工序在时间上严格排在该机器所有固定工序之后
2. **MoldReadyTimes**：确保动态工序等待模具从固定工序中释放
3. **CP-SAT AddNoOverlap**：确保动态工序之间不重叠

**真正需要修复的问题：**

1. **（必须修复）** `OrderByDescending` 是代码 bug，会导致 scheduleTime 取到"最晚"而非"最早"的 PlanStartTime
2. **（必须修复）** scheduleTime 可能早于 DateTime.Now，导致排出的动态工序 ApsStartTime 落在过去
3. **（建议修复）** 固定工序的输入数据重叠未检测，需要在 SchedulingEngine 中加警告日志
4. **（建议修复）** scheduleTime 截断到分钟，消除毫秒级舍入噪声
