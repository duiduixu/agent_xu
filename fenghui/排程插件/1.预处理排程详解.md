## 1. 前言：这篇文章是写给谁的

如果你已经看完了 `CpSatOptimizer` 的代码（或配套的 [CP-SAT 优化器详解教程](./CP-SAT优化器详解教程.md)），你会发现 CP-SAT 的输入是一个结构良好的 `SchedulingProblem` 对象，里面有 `Job` 列表、`Machine` 列表、各种约束状态……

但**这些数据不是凭空产生的**。在进入求解器之前，原始的宿主系统 JSON 数据需要经过一系列**清洗、过滤、转换、分类**操作，才能变成求解器可以理解的结构化输入。

**这就是预处理器的职责。**

本教程将以 `SchedulingPreprocessor.Prepare` 方法为主线，用小白也能看懂的语言，逐步解释：

- 原始工单数据是如何变成 `Job` 领域模型的
- 哪些工单被过滤掉了？为什么？
- "固定工序"和"动态工序"有什么区别？
- 机器锚点（Anchor）和模具就绪时间（MoldReadyTime）是怎么算出来的？
- 模具是怎么选择的？候选机器是怎么确定的？

## 2. 背景：预处理在整个排程流水线中的位置

在深入代码之前，先了解预处理在整个系统中处于什么位置。

### 排程流水线的"三剑客"

```
宿主系统 JSON 输入
        │
        ▼
┌───────────────────────────────┐
│ ① SchedulingPayloadNormalizer │  ← 输入归一化（JSON 类型转换）
│   字符串→数字、空值补全等        │
└───────────────┬───────────────┘
                │ SchedulingInput（已归一化）
                ▼
┌───────────────────────────────┐
│ ② SchedulingPreprocessor      │  ← ★ 本文档的主角 ★
│   Prepare()                   │    验证、过滤、分类、构建领域模型
└───────────────┬───────────────┘
                │ SchedulingProblem（结构化问题）
                ▼
┌───────────────────────────────┐
│ ③ SchedulingEngine            │
│   ├── HeuristicScheduler.Build│  ← 启发式初排
│   └── CpSatOptimizer.Optimize │  ← CP-SAT 精确优化
└───────────────────────────────┘
                │
                ▼
          ScheduleResult（排程结果）
```

### 预处理器的"输入 vs 输出"

| | 输入 | 输出 |
|---|------|------|
| **类型** | `SchedulingInput`（原始 DTO） | `SchedulingProblem`（领域模型） |
| **工单** | `PendingWorkOrder`（外部格式） | `Job`（内部模型）+ 分类标签 |
| **工单状态** | 混合在一起 | 分四类：动态Job / 固定工序 / 忽略 / 不可排产 |
| **机器** | 分散在 MpdRelation 中 | 去重后统一到 `Machines` 列表 |
| **约束状态** | 无 | 机器锚点 + 模具就绪时间 + 颜色策略 |

## 4. 逐步详解：7 个阶段走完预处理流程

### 阶段 1：输入非空校验

```csharp
SchedulingProblemValidator.ValidateInput(input);
```

**做了哪些检查**：

| 检查项 | 原因 |
|--------|------|
| `input` 不为 null | 最基本的防御 |
| `input.ScheduleConfig` 不为 null | 换模时间、全量重排开关的容器 |
| `input.Settings` 不为 null | 权重和 Horizon 的容器 |
| `input.WorkOrders` 不为 null | 工单列表 |
| `input.MpdRelations` 不为 null | 模具-产品-设备关系 |
| `input.ColorGroups` 不为 null | 颜色组规则 |
| `input.ColorSwitchRules` 不为 null | 颜色切换规则 |

> **设计理念**：这是"快速失败"（Fail-Fast）模式——发现问题立即抛异常，不让坏数据进入后续的复杂处理逻辑。

---

### 阶段 2：构建映射表

这是预处理的"地基"——后续所有过滤和转换都依赖这几个映射表。

#### 2a. 确定排程基准时间（scheduleTime）

```csharp
var scheduleTime = input.WorkOrders
    .Where(x => x.Status == WorkOrderStatus.PendingSchedule
             && x.PlanStartTime != null)
    .OrderByDescending(x => x.PlanStartTime)
    .FirstOrDefault(x => x.Status == WorkOrderStatus.PendingSchedule)
        ?.PlanStartTime ?? DateTime.Now;
```

**这段代码在说什么？**

在所有"待排产"状态的工单中，找到**最晚的计划开始时间**作为排程基准时间。

为什么要取"最晚"而不是"最早"？——因为所有未排产工单都应该从这个时间之后开始排产，最晚的计划开始时间确保所有工单的计划开始时间都小于等于排程基准时间。

**如果没有 PlanStartTime**：使用 `DateTime.Now`（当前时间）。

#### 2b. 颜色映射表（colorMap）

```csharp
var colorMap = input.ColorGroups
    .Where(x => !string.IsNullOrWhiteSpace(x.Color))
    .GroupBy(x => Normalize(x.Color))
    .ToDictionary(
        x => x.Key,
        x => x.OrderBy(y => y.Priority ?? int.MaxValue).First(),
        StringComparer.OrdinalIgnoreCase);
```

**这段代码在说什么？**

把颜色组规则按颜色名分组，同一颜色有多个规则时，取 **Priority 最小**（数字越小越优先，即颜色越浅越优先）的那条。

```
假设输入：
  颜色"象牙白" → GroupName="白色系", Priority=10
  颜色"IVORY WHITE" → GroupName="白色系", Priority=5
  颜色"象牙白" → GroupName="白色系", Priority=3

归一化后 "象牙白" 和 "IVORY WHITE" 都变成 "象牙白"（以首次出现为准？不）
```

**Normalize 做了什么？**
```csharp
private static string Normalize(string value)
{
    if (string.IsNullOrWhiteSpace(value)) return value;
    return value.Trim().ToUpperInvariant();
}
```
去除首尾空格 + 转大写。这确保 "象牙白" 和 " 象牙白 " 被视为同一颜色。

#### 2c. 产品-关系映射表（relationGroups）

```csharp
var relationGroups = input.MpdRelations
    .Where(IsUsableRelation)
    .GroupBy(x => Normalize(x.ProductId))
    .ToDictionary(x => x.Key, x => x.ToList(), StringComparer.OrdinalIgnoreCase);
```

**这段代码在说什么？**

从所有模具-产品-设备关系（MpdRelation）中，筛选出**可用的**关系，然后按产品 ID 分组。

**什么是"可用的"关系？**

```csharp
private static bool IsUsableRelation(MpdRelation relation) =>
    string.Equals(relation.MoldUseState, "正常", StringComparison.OrdinalIgnoreCase)
    && string.Equals(relation.DeviceUseState, "正常", StringComparison.OrdinalIgnoreCase)
    && !string.IsNullOrWhiteSpace(relation.ProductId)
    && !string.IsNullOrWhiteSpace(relation.MoldId)
    && !string.IsNullOrWhiteSpace(relation.DeviceId);
```

四个条件必须**全部满足**：

| 条件 | 含义 |
|------|------|
| `MoldUseState == "正常"` | 模具没有被维修或报废 |
| `DeviceUseState == "正常"` | 设备没有被维修或停用 |
| `ProductId` 非空 | 产品必须有 ID |
| `MoldId` 非空 | 模具必须有 ID |
| `DeviceId` 非空 | 设备必须有 ID |

**结果示例**：
```
relationGroups = {
    "PRODUCT-A": [
        { MoldId: "M001", DeviceId: "M1", ProduceCycleSeconds: 30, HolesNumber: 4 },
        { MoldId: "M002", DeviceId: "M1", ProduceCycleSeconds: 25, HolesNumber: 2 },
        { MoldId: "M001", DeviceId: "M2", ProduceCycleSeconds: 30, HolesNumber: 4 },
    ],
    "PRODUCT-B": [
        { MoldId: "M003", DeviceId: "M1", ... },
        { MoldId: "M003", DeviceId: "M3", ... },
    ]
}
```

#### 2d. 机器字典（machines）

```csharp
var machines = new Dictionary<string, Machine>(StringComparer.OrdinalIgnoreCase);
foreach (var relation in input.MpdRelations.Where(IsUsableRelation))
{
    machines[relation.DeviceId] = new Machine(
        relation.DeviceId, relation.DeviceName, relation.DeviceModel, relation.Tonnage);
}
```

**这段代码在说什么？**

从所有有效关系中提取设备列表，按 DeviceId 去重后存为字典。后续阶段 3h 还会为遗漏的设备补建对象（防御性代码）。

### 阶段 3：遍历工单进行过滤和分类

这是整个预处理最核心、最复杂的阶段。我们按照代码的实际执行顺序，逐个检查点来分析。

```
对每个工单，依次经过以下"关口"：

  workOrder 进入
      │
      ▼
  ┌─────────┐
  │ 3a. 完工? │──是──▶ 跳过（不参与排程）
  └────┬────┘
       │否
       ▼
  ┌──────────┐
  │ 3b. 锁定? │──是──▶ 加入 ignoredOrders
  └────┬─────┘
       │否
       ▼
  ┌─────────────────┐
  │ 3c. 有有效关系?  │──否──▶ 加入 unschedulableOrders
  └────┬────────────┘      ("未找到产品对应的有效模具-设备关系")
       │是
       ▼
  ┌─────────────────┐
  │ 3d. 颜色已配置?  │──否──▶ 加入 unschedulableOrders
  └────┬────────────┘      ("颜色未配置颜色组优先级")
       │是
       ▼
  ┌────────────┐
  │ 3e. 构建模具 │
  │    候选列表  │
  └────┬───────┘
       │
       ▼
  ┌───────────────┐
  │ 3f. 选择模具   │──失败──▶ 加入 unschedulableOrders
  └────┬──────────┘          ("工单未匹配到可用模具")
       │成功
       ▼
  ┌──────────────────┐
  │ 3g. 确定候选设备池│──为空──▶ 加入 unschedulableOrders
  └────┬─────────────┘         ("工单未匹配到可生产设备")
       │非空
       ▼
  ┌──────────────┐
  │ 3h. 补建设备   │（防御性代码）
  └────┬─────────┘
       │
       ▼
  ┌──────────────────┐
  │ 3i. 剩余数量=0?   │──是──▶ 跳过
  └────┬─────────────┘
       │否
       ▼
  ┌───────────────┐
  │ 3j. 判断属性   │──▶ shouldOptimize? 还是 shouldAnchor?
  └────┬──────────┘
       │
       ▼
  ┌────────────────┐
  │ 3k. 计算交期和  │
  │    生产时长     │
  └────┬───────────┘
       │
       ▼
  ┌──────────────┐
  │ 3l. 构建 Job  │
  └────┬─────────┘
       │
       ▼
  ┌──────────────┐
  │ 3m. 分类存储  │──▶ fixedOperations 或 dynamicJobs
  └──────────────┘
```

#### 3a. 已完工工单跳过

```csharp
if (workOrder.Status == WorkOrderStatus.Completed)
{
    continue;
}
```

完工的工单已经不需要排程了，直接跳过。

#### 3b. 锁定工单忽略

```csharp
if (workOrder.IsLocked == true)
{
    ignoredOrders.Add(workOrder);
    continue;
}
```

锁定的工单不参与排程（可能是管理员手动锁定的），但记录下来以便后续展示给用户。

#### 3c~3d. 数据完整性检查

```csharp
// 3c. 检查产品是否有对应的模具-设备关系
if (!relationGroups.TryGetValue(Normalize(workOrder.ProductId), out var relations) || relations.Count == 0)
{
    unschedulableOrders.Add(new UnschedulableOrder(workOrder, "未找到产品对应的有效模具-设备关系。"));
    continue;
}

// 3d. 检查颜色是否已配置
if (!colorMap.TryGetValue(Normalize(workOrder.ProductColor), out var colorRule))
{
    unschedulableOrders.Add(new UnschedulableOrder(workOrder, "颜色未配置颜色组优先级。"));
    continue;
}
```

这两个检查确保排程所需的基础数据是完整的。如果有缺失，将工单标记为"不可排产"并附带原因，供前端展示给用户。

#### 3e. 构建模具候选列表（MoldCandidate）

```csharp
var moldCandidates = relations
    .GroupBy(x => x.MoldId, StringComparer.OrdinalIgnoreCase)
    .Select(group => new MoldCandidate(
        group.Key,
        group.First().ProduceCycleSeconds ?? 0,        // 生产周期（秒）
        Math.Max(1, group.First().HolesNumber ?? 1),   // 模穴数（至少为 1）
        group.Select(x => x.DeviceId)
             .Distinct(StringComparer.OrdinalIgnoreCase)
             .ToList()))                                // 该模具可用的设备列表
    .ToList();
```

**这段代码在说什么？**

将属于该产品的所有有效 MpdRelation 按模具 ID 分组，每组汇总成一个 `MoldCandidate`：

| 字段 | 来源 | 含义 |
|------|------|------|
| `MoldId` | `group.Key` | 模具 ID |
| `ProduceCycleSeconds` | `MpdRelation.ProduceCycleSeconds` | 该模具生产该产品的单次周期（秒） |
| `HolesNumber` | `MpdRelation.HolesNumber`（至少 1） | 模穴数（一次注塑可产出几个产品） |
| `DeviceIds` | 该模具上所有关系的 DeviceId 去重 | 该模具可以在哪些设备上使用 |

**示例**：
```
产品 "PRODUCT-A" 的有效关系：
  M001 → [M1, M2]  周期=30秒, 穴数=4
  M002 → [M1]       周期=25秒, 穴数=2
  M001 → [M3]       周期=30秒, 穴数=4（同模具不同设备）

构建的 MoldCandidate 列表：
  MoldCandidate("M001", 30, 4, ["M1", "M2", "M3"])
  MoldCandidate("M002", 25, 2, ["M1"])
```

#### 3f. 选择模具（SelectMold）

```csharp
var selectedMold = SelectMold(workOrder, moldCandidates);
```

详见 [5.1 SelectMold：模具选择策略](#51-selectmold模具选择策略)。

关键逻辑：优先用工单指定的模具（`workOrder.MoldId`），未指定时选**设备数最多**的模具（生产灵活性最大）。

#### 3g. 确定候选设备池

```csharp
var candidateDevices = moldCandidates
    .SelectMany(x => x.DeviceIds)
    .Distinct(StringComparer.OrdinalIgnoreCase)
    .OrderBy(x => x, StringComparer.OrdinalIgnoreCase)
    .ToList();
```

**关键理解**：候选设备池来自**该产品所有有效模具的全部设备**，而不是只限定在 `selectedMold` 的设备范围内。

这意味着：即使模具 A 被选为"默认模具"，Job 仍然可以在模具 B 的设备上生产——这对 CP-SAT 的优化很重要，因为求解器可以在更大范围内寻找最优分配。

**为什么这样设计？** 参见 [7. 关键设计决策](#7-关键设计决策)。

#### 3h. 防御性补建 Machine 对象

```csharp
foreach (var deviceId in candidateDevices.Where(id => !machines.ContainsKey(id)))
{
    machines[deviceId] = new Machine(deviceId, deviceId);
}
```

如果候选设备中有在阶段 2d 未被收录的设备（比如来自不同的 MpdRelation 批次），则补充创建。这确保后续验证不会因为"Job 引用的机器不存在"而失败。

#### 3i. 计算剩余待产数量

```csharp
var remainingQuantity = Math.Max(0,
    (workOrder.ScheduleQuantity ?? 0)
    - Math.Max(workOrder.CompleteQuantity ?? 0, workOrder.FinishQuantity ?? 0));

if (remainingQuantity == 0)
{
    continue; // 已全部完成
}
```

`remainingQuantity` = 计划数量 - max(已完成数量, 已完工数量)

如果剩余为 0，说明该工单已经全部完成，不再需要排程。

#### 3j. 判断工单属性：优化 vs 锚定

```csharp
var shouldOptimize = ShouldOptimize(workOrder, input.ScheduleConfig);
var shouldAnchor = !shouldOptimize
                   && !string.IsNullOrWhiteSpace(workOrder.DeviceId)
                   && workOrder.Status != WorkOrderStatus.PendingSchedule;
```

详见 [5.4 ShouldOptimize：优化/锚定决策](#54-shouldoptimize优化锚定决策)。

**一句话总结**：
- `shouldOptimize = true`：工单可以自由选择机器和时间（参与优化）
- `shouldAnchor = true`：工单固定在当前机器上，不可移动
- 两者都 false：工单锁定在 DeviceId 上，但不"锚定"（仍在当前机器但可调时间）

**确定候选机器列表**：
```csharp
var eligibleMachines = shouldAnchor
    || (!string.IsNullOrWhiteSpace(workOrder.DeviceId)
        && workOrder.Status != WorkOrderStatus.PendingSchedule)
    ? new[] { workOrder.DeviceId }.ToList()   // 只有一台机器
    : candidateDevices;                        // 全部候选设备
```

- **锚定或非待排产状态**：只能在当前设备（DeviceId）上生产
- **待排产状态（且非全量重排）**：可在全部候选设备池中选择

#### 3k. 计算交期和预计生产时长

```csharp
var dueTime = workOrder.DueDateTime
    ?? scheduleTime.AddMinutes(input.Settings.HorizonMinutes);

var durationMinutes = ComputeDurationMinutes(
    remainingQuantity,
    selectedMold.ProduceCycleSeconds,
    selectedMold.HolesNumber);
```

**交期（DueMinutes）**：
- 如果工单有 DueDateTime：交期 = `(DueDateTime - scheduleTime)` 的分钟数
- 如果工单没有 DueDateTime：默认交期 = Horizon（48 小时后），即"不紧迫"

**生产时长（DurationMinutes）**：详见 [5.2 ComputeDurationMinutes](#52-computedurationminutes生产时长计算)。

公式：`max(1, ceil(剩余数量 × 周期秒 / 60 / 模穴数))`

### 阶段 4：构建机器锚点状态

```csharp
var orderedFixedOperations = fixedOperations
    .OrderBy(x => x.StartMinutes)
    .ThenBy(x => x.EndMinutes)
    .ToList();

var machineAnchors = orderedFixedOperations
    .GroupBy(x => x.MachineId, StringComparer.OrdinalIgnoreCase)
    .ToDictionary(
        x => x.Key,
        x =>
        {
            var tail = x.OrderByDescending(y => y.EndMinutes).First();
            return new MachineAnchorState(tail.EndMinutes, tail.Job);
        },
        StringComparer.OrdinalIgnoreCase);
```

**这段代码在说什么？**

1. 将所有固定工序按开始时间 → 结束时间排序
2. 按机器 ID 分组
3. 对每组，找到**结束时间最晚**的固定工序
4. 记录该工序的结束时间和 Job，作为该机器的"锚点"

**锚点的作用**：后续所有动态 Job 在这台机器上的开始时间，必须 ≥ 锚点结束时间（+ 换线时间）。

```
示例：机器 M1 上有 3 个固定工序
  工序A: [0────30]
  工序B: [30──────80]
  工序C: [80─────────120]

MachineAnchorState("M1") = { ReadyMinutes: 120, LastJob: 工序C的Job }
→ 后续动态 Job 最早只能从 120 分钟（+ 换线时间）开始
```

### 阶段 5：构建模具就绪时间

```csharp
var moldReadyTimes = orderedFixedOperations
    .GroupBy(x => x.Job.MoldId, StringComparer.OrdinalIgnoreCase)
    .ToDictionary(
        x => x.Key,
        x => x.Max(y => y.EndMinutes),
```

**这段代码在说什么？**

1. 将所有固定工序按模具 ID 分组
2. 对每组，取**所有工序的最晚结束时间**
3. 作为该模具的"就绪时间"

```
示例：模具 M001 被以下固定工序使用
  机器 M1 上：工序A 使用 M001 [0────60]
  机器 M2 上：工序B 使用 M001 [30────90]

MoldReadyTime["M001"] = 90（取最晚结束时间）
→ 任何使用模具 M001 的动态 Job，最早只能从 90 分钟开始
```

## 5. 关键子方法详解

### 5.1 SelectMold：模具选择策略

```csharp
private static MoldCandidate? SelectMold(
    PendingWorkOrder workOrder,
    IReadOnlyList<MoldCandidate> candidates)
{
    // 策略 1：工单指定了模具，直接匹配
    if (!string.IsNullOrWhiteSpace(workOrder.MoldId))
    {
        return candidates.FirstOrDefault(x =>
            string.Equals(x.MoldId, workOrder.MoldId, StringComparison.OrdinalIgnoreCase));
    }

    // 策略 2：未指定模具时，按规则自动选择
    return candidates
        .OrderByDescending(x => x.DeviceIds.Count)     // ① 候选设备数 ↓
        .ThenBy(x => x.ProduceCycleSeconds)            // ② 生产周期 ↑
        .ThenBy(x => x.MoldId, StringComparer.OrdinalIgnoreCase)  // ③ 模具 ID 字典序
        .FirstOrDefault();
}
```

**选择策略可视化**：

```
工单是否指定了 MoldId？
  ├── 是 → 查找匹配的模具（精确匹配）
  │        → 找到？返回
  │        → 未找到？返回 null → 工单标记为"不可排产"
  │
  └── 否 → 自动选择，优先级：
           ① 候选设备数最多的模具（灵活性最大）
           ② 生产周期最短的模具（效率最高）
           ③ 模具 ID 字典序（稳定性保证：相同条件下总选同一个）
```

**为什么优先选设备数多的模具？**

设备数多的模具可以让 CP-SAT 有更大的分配自由度。如果选了一个只能在一台设备上用的模具，求解器就失去了在其他设备上排产的可能性。

### 5.2 ComputeDurationMinutes：生产时长计算

```csharp
private static int ComputeDurationMinutes(
    int scheduleQuantity,      // 剩余待产数量
    int produceCycleSeconds,   // 单次注塑周期（秒）
    int holesNumber)           // 模穴数
{
    if (scheduleQuantity <= 0) return 0;

    var cavities = Math.Max(1, holesNumber);       // 模穴数至少为 1
    var cycleSeconds = Math.Max(1, produceCycleSeconds);  // 周期至少为 1 秒
    return Math.Max(1, (int)Math.Ceiling(
        scheduleQuantity * cycleSeconds / 60d / cavities));
}
```

**公式推导**：

```
每次注塑生产的产品数 = cavities（模穴数）
每次注塑耗时 = cycleSeconds（秒）

生产 N 个产品需要的注塑次数 = ceil(N / cavities)
总耗时秒数 = 注塑次数 × cycleSeconds
总耗时分钟数 = 总耗时秒数 / 60

合在一起：ceil(N × cycleSeconds / 60 / cavities)
```

**直观理解**：一个模具有 4 个穴，一次可以同时注塑 4 个产品。生产 1000 个产品需要注塑 250 次。如果每次注塑 30 秒，总耗时 = 250 × 30 = 7500 秒 = 125 分钟。

### 5.3 CreateFixedOperation：固定工序创建

```csharp
private static ScheduledOperation CreateFixedOperation(
    DateTime scheduleTime, PendingWorkOrder workOrder, Job job)
{
    // 确定开始时间：优先计划开始时间 → 实际开始时间 → 排程基准时间
    var startTime = workOrder.PlanStartTime
        ?? workOrder.ActualStartTime
        ?? scheduleTime;

    // 如果工单在实际排程前已开始，从当前时刻开始计算
    if (workOrder.ActualStartTime.HasValue
        && workOrder.ActualStartTime.Value <= scheduleTime)
    {
        startTime = scheduleTime;
    }

    // 确定结束时间：优先计划结束时间 → 开始时间+生产时长
    var endTime = workOrder.PlanEndTime
        ?? startTime.AddMinutes(job.DurationMinutes);
    if (endTime < scheduleTime)
    {
        endTime = scheduleTime;
    }

    // 转换为相对分钟数
    var startMinutes = Math.Max(0,
        (int)Math.Ceiling((startTime - scheduleTime).TotalMinutes));
    var endMinutes = Math.Max(startMinutes,
        (int)Math.Ceiling((endTime - scheduleTime).TotalMinutes));

    return new ScheduledOperation(
        job, workOrder.DeviceId, startMinutes, endMinutes,
        Math.Max(0, endMinutes - job.DueMinutes),
        true);  // IsFixed = true
}
```

**开始时间的优先级**：

```
PlanStartTime（计划开始）
  ↓ 如果为 null
ActualStartTime（实际开始）
  ↓ 如果为 null
scheduleTime（排程基准时间）
```

**特殊情况处理**：

| 情况 | 处理方式 |
|------|---------|
| `ActualStartTime ≤ scheduleTime` | 工单已经在进行中，从 scheduleTime 起算剩余部分 |
| `endTime < scheduleTime` | 异常数据修正，结束时间至少为 scheduleTime |
| `startMinutes` 可能为负 | 用 `Math.Max(0, ...)` 修正为 0 |

### 5.4 ShouldOptimize：优化/锚定决策

```csharp
private static bool ShouldOptimize(
    PendingWorkOrder workOrder, ScheduleConfig config)
{
    // 全量重排模式：所有未开始的工单都参与优化
    if (config.IsFullRescheduleEnabled)
    {
        return !IsStarted(workOrder);
    }

    // 非全量重排模式：仅"待排产"状态的工单参与优化
    return workOrder.Status == WorkOrderStatus.PendingSchedule;
}
```

**两种模式对比**：

| | 全量重排模式 | 增量排程模式（默认） |
|---|-------------|---------------------|
| **触发条件** | `ScheduleConfig.FullReschedule == "是"` | 其他情况 |
| **参与优化的工单** | 所有未开始的工单 | 仅"待排产"状态的工单 |
| **已开始的工单** | 锚定在固定机器上 | 锚定在固定机器上 |
| **非待排产但未开始的工单** | 参与优化！ | **也锚定**（不参与优化） |

**"非待排产但未开始"的工单是什么？**

比如状态为 `Released`（已下达）、`SetupInProgress`（调机中）的工单。在增量模式下，这些工单虽然没开始，但因为已经分配了资源和时间，不应该被重新挪动——它们被锚定在原地。

### 5.5 IsStarted：工单状态判断

```csharp
private static bool IsStarted(PendingWorkOrder workOrder)
{
    // 有实际开始时间 = 必然已开始
    if (workOrder.ActualStartTime.HasValue)
    {
        return true;
    }

    // 根据状态判断
    return workOrder.Status is WorkOrderStatus.SetupInProgress
        or WorkOrderStatus.ChangeoverInProgress
        or WorkOrderStatus.AwaitingFirstInspection
        or WorkOrderStatus.Interrupted
        or WorkOrderStatus.Paused
        or WorkOrderStatus.InProduction;
}
```

**判断"已开始"的两个标准**：

1. **有实际开始时间**：`ActualStartTime != null`——这是最直接的证据
2. **处于生产流程状态**：以下 6 种状态表示工单已经在产线上流动

| 状态 | 含义 | 为什么算"已开始" |
|------|------|------------------|
| `SetupInProgress` | 调机中 | 正在为生产做准备 |
| `ChangeoverInProgress` | 转产中 | 正在切换模具/物料 |
| `AwaitingFirstInspection` | 待首检 | 已开始生产，等待首个产品检验 |
| `Interrupted` | 中断 | 生产中断但未取消 |
| `Paused` | 暂停 | 生产暂停但未取消 |
| `InProduction` | 生产中 | 正在生产 |

**不算"已开始"的状态**：`PendingSchedule`（待排产）、`Released`（已下达）、`Unknown`（未知）、`Completed`（已完工）。

### 5.6 IsUsableRelation：关系可用性过滤

```
            ┌─────────────────────┐

                      │是
            ┌───────────────────────┐
            │ DeviceUseState=="正常" │──否──▶ 不可用（设备维修/停用）
            └─────────┬─────────────┘
                      │是
            ┌────────────────┐
            │ ProductId 非空  │──否──▶ 不可用（数据不完整）
            └────────┬───────┘

            ┌─────────────┐

            ┌───────────────┐
```

## 7. 关键设计决策

### 决策 1：设备池来自全部有效模具，而非单一模具

```csharp
// 3g. 候选设备池 = 该产品所有有效模具的设备并集
var candidateDevices = moldCandidates
    .SelectMany(x => x.DeviceIds)
    .Distinct()
    .ToList();
```

**为什么？**

在早期版本中，设备池被限定在选定模具（selectedMold）的设备上。这限制了求解器的搜索空间。改为全部有效模具的设备池后：

- 启发式排程器可以在更大范围内分配 Job
- CP-SAT 有更多可选方案，可能找到更优解
- `selectedMold` 退化为"默认模具"，仅用于**生产时长计算**

### 决策 2：先做设备分配，再做设备内重排

这是 `HeuristicScheduler` 的两阶段策略（详见启发式排程文档），预处理器通过以下方式支持此策略：

- `EligibleMachines` 存储候选设备列表
- `MachineAnchors` 和 `MoldReadyTimes` 提供时间约束

### 决策 3：颜色组规则取最低 Priority

```csharp
x => x.OrderBy(y => y.Priority ?? int.MaxValue).First()
```

如果同一颜色有多个颜色组规则（比如不同的管理员配置了不同的优先级），取 Priority 最小的。小 Priority = 更优先（颜色越浅越优先），这在注塑行业是最安全的选择（浅色先生产）。

## 8. 常见疑问解答（FAQ）

### Q1：为什么有的工单被标记为"不可排产"而不是直接抛异常？

因为一个工单不可排产不应该阻止整个排程流程。排程是一个批量操作，应该尽可能地处理能排的工单，同时告知用户哪些工单有问题、为什么有问题。

`unschedulableOrders` 列表会随 `SchedulingProblem` 一起传递给调度引擎，最终返回给前端展示。

### Q2：固定工序和动态 Job 的区别是什么？

| | 固定工序 | 动态 Job |
|---|---------|---------|
| **是否可移动** | 不可移动 | 可以自由分配 |
| **机器** | 固定在指定机器 | 可在候选机器中选择 |
| **时间** | 时间已确定 | 时间由求解器决定 |
| **来源** | 已开始/锚定的工单 | 待排产/未开始的工单 |
| **参与优化** | 不参与，只作为约束 | 参与，由求解器分配 |
| **数据结构** | `ScheduledOperation` (IsFixed=true) | `Job`（等待被调度） |

### Q3：MachineAnchor 和 MoldReadyTime 有什么区别？

两者都是"动态 Job 的最早开始时间约束"，但维度不同：

| | MachineAnchor | MoldReadyTime |
|---|--------------|---------------|
| **约束维度** | 机 器 | 模 具 |
| **含义** | 机器上最后一个固定工序结束前，不能开始新 Job | 模具被固定工序占用结束前，不能用该模具 |
| **影响范围** | 该机器上的所有动态 Job | 使用该模具的所有动态 Job（可能在多台机器上） |
| **计算方式** | 每台机器的最后一个固定工序 | 每套模具的所有固定工序的最晚时间 |

### Q4：全量重排和增量排程的核心区别是什么？

核心区别在于**哪些工单被重新考虑**：

```
增量排程（默认）：
  PendingSchedule 工单 → 参与优化
  Released/SetupInProgress/... 工单 → 锚定不动
  已开始工单 → 锚定不动

全量重排：
  未开始工单 → 参与优化
  已开始工单 → 锚定不动
```

全量重排适用于以下场景：
- 工厂换班、周末后重新规划
- 大量工单完成或取消，需要全局重排
- 管理者手动触发"一键排程"

### Q5：为什么 MoldCandidate 中的 HolesNumber 和 ProduceCycleSeconds 都至少要等于 1？

防止除零错误。`ComputeDurationMinutes` 中使用了 `scheduleQuantity * cycleSeconds / cavities` 的计算公式，如果 cavities（模穴数）为 0 会导致除零异常。

### Q6：Normalize 为什么用 `ToUpperInvariant` 而不是 `ToLowerInvariant`？

两种方式都可以达到"忽略大小写"的目的。用 `ToUpperInvariant` 是习惯性选择——在土耳其语等特定文化中，`ToUpperInvariant` 比 `ToLowerInvariant` 更安全（避免著名的"Turkish İ problem"）。
