# 插件权限收敛技术开发方案

## 1. 背景与结论

当前插件模块支持插件程序集内直接声明 `IDynamicApiController`，插件安装并启用后即可把接口挂到主系统请求管道中生效。  
这种模式虽然灵活，但存在以下明显问题：

1. **权限过大**：插件可以直接暴露 HTTP 接口，实际拥有了“扩展宿主 API 面”的能力。
2. **安全边界弱**：插件可自行决定路由、入参、返回值和调用时机，宿主难以统一鉴权、审计、限流、幂等和参数校验。
3. **接口冲突风险**：不同插件之间可能出现路由冲突，插件与主系统接口也可能冲突。
4. **治理复杂**：宿主无法从业务层明确知道“某个接口调用了哪个插件、插件做了什么、是否允许调用”。
5. **升级不可控**：插件接口一旦成为外部依赖，后续插件实现变更会直接影响平台 API 稳定性。

**结论：你提出的方案更好。**

即：

- **插件只负责算法/能力实现**
- **固定 HTTP 接口只保留在主项目**
- **主项目接口通过 `pluginId` 或“能力编码 + pluginId”选择具体插件实现**

这是一种更适合工业平台和企业级系统的方案，本质上是把插件从“可扩展 API”收敛为“可扩展能力提供者（Capability Provider）”。

---

## 2. 目标架构

### 2.1 改造目标

将现有插件模型从：

> 插件 = 元数据 + DI + 生命周期 + 动态接口

改造为：

> 插件 = 元数据 + DI + 生命周期 + 受控能力实现

宿主负责：

- 暴露固定 API
- 鉴权
- 参数校验
- 审计日志
- 限流
- 统一异常处理
- 插件选择策略
- 插件启停与路由治理

插件负责：

- 实现宿主预先定义好的算法契约
- 注册自身能力实现
- 在受控容器中被宿主调用

### 2.2 目标调用链

```text
客户端
  -> 主项目固定 API
      -> 参数校验 / 鉴权 / 审计 / 限流
          -> 根据 pluginId 查找已启用插件
              -> 根据能力契约解析插件服务
                  -> 执行插件算法
                      -> 返回统一结果
```

### 2.3 核心原则

1. **插件不得直接暴露 HTTP 路由**
2. **插件只实现能力接口，不实现控制器接口**
3. **宿主只按白名单能力接口调用插件**
4. **主项目对外 API 地址固定，不因插件而变化**
5. **同一能力允许多个插件实现，但调用入口仍由宿主统一治理**

---

## 3. 与现方案对比

| 维度 | 现方案：插件直接出接口 | 新方案：宿主固定接口 + 插件只做能力 |
|---|---|---|
| 安全性 | 低，插件可扩展 API 面 | 高，插件仅在宿主受控范围内执行 |
| 路由冲突 | 高 | 极低 |
| 鉴权一致性 | 难统一 | 容易统一 |
| 限流/审计 | 需插件自觉配合 | 宿主统一处理 |
| API 稳定性 | 易受插件影响 | 稳定 |
| 平台治理 | 弱 | 强 |
| 插件开发自由度 | 高但风险大 | 适中且可控 |
| 长期可维护性 | 一般 | 更好 |

**结论**：对于 IotPlatform 这类平台型系统，推荐采用新方案作为主线架构。

## 5. 目标能力模型设计

### 5.1 插件能力分层

建议把插件接口收敛为以下三层：

1. **插件元数据层**
   - 插件 ID、名称、版本、类型、描述

2. **插件生命周期层**
   - `IPlugin`
   - `IPluginLifecycle`

3. **插件能力层**
   - 宿主预定义的能力契约接口
   - 例如 APS 场景定义 `IApsSchedulingProvider`

### 5.2 能力契约示例

建议在 `Plugin.Contract` 中增加能力接口，例如：

```csharp
namespace Plugin.Contract.Capabilities;

public interface IApsSchedulingProvider
{
    Task<ApsSchedulingResult> RunAsync(ApsSchedulingRequest request, CancellationToken cancellationToken = default);
}
```

同时建议把宿主与插件共享的算法输入输出 DTO 也放到契约层，例如：

- `ApsSchedulingRequest`
- `ApsSchedulingResult`
- `ApsSchedulingJob`
- `ApsSchedulingMachine`

**原则**：  
插件与宿主之间只共享“契约 DTO”，不共享控制器模型，不共享 MVC 语义，不共享 HTTP 相关特性。

### 5.3 能力声明方式

推荐新增显式能力声明机制，避免宿主盲扫任意类型。

建议二选一：

#### 方案 A：启动类显式注册

```csharp
public interface IPluginServiceRegistrar
{
    void Register(IServiceCollection services);
}
```

由插件启动类在 `ConfigureServices` 中注册：

```csharp
services.AddSingleton<IApsSchedulingProvider, ApsSchedulingProvider>();
```

#### 方案 B：新增能力描述接口

```csharp
public interface IPluginCapabilityDescriptor
{
    IReadOnlyCollection<string> CapabilityCodes { get; }
}
```

例如 APS 插件声明：

- `aps:scheduling`

**推荐**：A + B 组合使用。  
即：

- **A** 负责 DI 注册
- **B** 负责宿主识别“这个插件提供了哪些能力”

## 6. 主项目插件模块改造方案

## 6.1 总体改造方向

插件模块应从“动态路由加载器”改为“能力加载器”。

### 当前应逐步废弃的能力

以下能力建议标记为弃用并最终删除：

- `PluginRouteDispatcherMiddleware`
- `UsePluginRouteDispatcher()`
- `PluginRouteDescriptor`
- `PluginRouteMatch`
- `PluginManager` 中的 HTTP 路由发现与匹配逻辑
- 对 `IDynamicApiController` 的插件扫描

### 保留并强化的能力

- 插件安装
- 插件启用
- 插件禁用
- 插件卸载
- 插件服务容器隔离
- 受控服务解析
- 生命周期回调
- 插件状态持久化

## 6.3 PluginManager 改造

### 当前职责保留

- 维护 `_plugins`
- 插件扫描/安装/启停/卸载
- 插件 `ServiceProvider` 生命周期管理
- `GetPluginService<TService>(pluginId)` 受控解析

### 新增职责

#### 1. 能力查询

建议新增：

```csharp
public TService? GetRequiredPluginCapability<TService>(string pluginId) where TService : class
```

或：

```csharp
public bool TryGetPluginCapability<TService>(string pluginId, out TService? service) where TService : class
```

#### 2. 能力校验

在启用插件时校验：

- 插件是否实现 `IPlugin`
- 插件是否声明能力编码
- 插件是否注册了声明的能力接口

#### 3. 能力元数据缓存

建议为 `PluginEntry` 新增：

- `CapabilityCodes`
- `ProvidedServiceTypes`

例如：

```csharp
public sealed class PluginEntry
{
    public IReadOnlyCollection<string> CapabilityCodes { get; internal set; } = Array.Empty<string>();
}
```

### 建议移除的逻辑

- `DiscoverHttpRoutes(Assembly assembly)`
- `MatchHttpRoute(...)`
- 所有 `IDynamicApiController` 扫描与路由分发逻辑

## 6.4 插件启用时的校验策略

启用插件时建议执行以下校验链：

1. 读取 `plugin.json`
2. 加载主程序集
3. 校验 `startupClass`
4. 实例化 `IPlugin`
5. 构建插件专属 `ServiceProvider`
6. 读取能力声明
7. 校验能力实现是否存在
8. 将能力元数据写入内存缓存
9. 更新插件状态为 `Enabled`

### 失败处理

任一校验失败：

- 不允许启用
- 状态置为 `Error`
- 输出明确日志
- 不暴露任何能力给宿主

## 6.6 插件白名单与能力授权

为了避免“只要实现了接口就能被调用”，建议增加一层配置化授权。

### 建议新增配置

在宿主配置中增加：

```json
{
  "Plugin": {
    "AllowedCapabilities": {
      "aps:scheduling": [
        "com.iotplatform.plugin.aps-scheduling"
      ]
    }
  }
}
```

### 用途

即使某个插件声明了 `aps:scheduling` 能力，宿主仍需校验：

1. 该能力是否允许插件化
2. 当前插件 ID 是否在白名单内

这样可以防止“恶意插件冒充已知能力接口”。

## 7.4 算法能力实现类改造

建议新增：

```text
Capabilities\ApsSchedulingProvider.cs
```

示例职责：

```csharp
public sealed class ApsSchedulingProvider : IApsSchedulingProvider
{
    private readonly SchedulingEngine _engine;

    public ApsSchedulingProvider(SchedulingEngine engine)
    {
        _engine = engine;
    }

    public Task<ApsSchedulingResult> RunAsync(ApsSchedulingRequest request, CancellationToken cancellationToken = default)
    {
        var result = _engine.Run(request);
        return Task.FromResult(result);
    }
}
```

### 说明

原先散落在 `ApsSchedulingAppService` 中的编排逻辑，应迁移到：

- `SchedulingEngine`
- `ApsSchedulingProvider`

控制器相关职责不得保留在插件中。

## 7.6 plugin.json 是否需要调整

`plugin.json` 仍然需要保留，因为宿主仍通过它发现插件。  
但其职责应只保留为：

- 标识插件
- 指向主程序集
- 指向启动类

**不再承载接口路由语义。**

如需声明插件能力，推荐有两种方式：

### 方式 A：仍通过程序集反射读取

宿主在启用插件时读取 `IPluginCapabilityDescriptor`

### 方式 B：在 `plugin.json` 中增加 `capabilities`

例如：

```json
{
  "id": "com.iotplatform.plugin.aps-scheduling",
  "name": "注塑生产排程",
  "version": "1.0.0",
  "type": "BackgroundService",
  "assembly": "Plugin.ApsScheduling.dll",
  "startupClass": "Plugin.ApsScheduling.PluginStartup",
  "capabilities": [
    "aps:scheduling"
  ]
}
```

**推荐**：`plugin.json` + 启动类双校验。  
即：

- `plugin.json` 声明能力
- 启动类也声明能力
- 宿主启用时校验二者一致

这样可以提升可观测性与一致性。

## 9. 安全治理方案

## 9.1 安全边界

改造后要实现的边界是：

- 插件可以被**调用**
- 插件不能自己**决定暴露什么 HTTP 接口**

## 9.2 具体安全措施

### 1. 禁止插件路由注入

启用插件时：

- 不再扫描 `IDynamicApiController`
- 不再注册插件路由
- 不再允许 `RouteAttribute` 决定宿主 API

### 2. 能力白名单

宿主只允许解析明确列入白名单的能力接口。

### 3. 插件 ID 白名单

对关键能力（如排程、控制、协议）增加插件 ID 白名单。

### 4. 输入校验前置

所有外部请求必须先在宿主完成参数校验，插件只接收规范化后的契约对象。

### 5. 审计与监控

记录以下信息：

- 调用时间
- 调用用户
- 插件 ID
- 能力编码
- 执行耗时
- 成功/失败
- 异常摘要

### 6. 超时控制

宿主调用插件时建议增加超时控制，避免算法长时间阻塞：

- 可使用 `CancellationToken`
- 或统一设置算法执行超时时间

### 7. 资源配额

后续若插件能力继续扩大，可再增加：

- 并发限制
- 内存配额
- CPU 配额

## 10. 兼容性与迁移策略

## 10.1 迁移策略建议

建议分三阶段执行：

### 阶段一：兼容过渡

1. 保留现有插件安装/启停机制
2. 新增能力契约与宿主桥接 API
3. 新插件只允许走能力模式
4. 老插件动态接口功能标记为“废弃”

### 阶段二：默认禁用插件动态接口

增加配置：

```json
{
  "Plugin": {
    "EnableDynamicApiFromPlugin": false
  }
}
```

默认关闭。  
仅在开发环境临时调试时允许显式开启。

### 阶段三：彻底删除动态接口能力

移除：

- 插件路由扫描
- 插件路由分发中间件
- 插件动态接口文档

## 10.2 对已有插件的影响

### 受影响插件类型

凡是依赖以下能力的插件都会受影响：

- `IDynamicApiController`
- 插件内 `Route`
- 插件内 `HttpGet/HttpPost`

### 不受影响插件类型

以下插件基本可平滑迁移：

- 纯算法插件
- 后台任务插件
- 协议解析插件
- 数据转换插件

## 12. 验收标准

## 12.1 主项目验收

1. 插件启用后，**不会新增任何插件自定义 HTTP 路由**
2. 宿主固定接口可通过 `pluginId` 调用指定插件
3. 未启用插件不可被调用
4. 未授权能力不可被调用
5. 插件异常能被宿主统一捕获并记录
6. 不同插件实现同一能力时，不会发生路由冲突

## 12.2 插件项目验收

1. `InjectionApsCpSatDemo` 中不再包含 `IDynamicApiController`
2. 插件仍可被安装、启用、禁用、卸载
3. 通过主项目固定接口传入 `pluginId` 后可以成功调用算法
4. 更换另一个同能力插件时，宿主 API 无需变化

## 13. 风险与注意事项

## 13.1 契约膨胀风险

如果每个业务都往 `Plugin.Contract` 增加大量 DTO，契约层会快速膨胀。  
建议按能力域拆分命名空间，例如：

- `Plugin.Contract.Aps`
- `Plugin.Contract.Protocol`
- `Plugin.Contract.Transform`

## 13.2 宿主与插件版本兼容

当契约 DTO 变化时，要明确版本策略。  
建议：

- `Plugin.Contract` 采用语义化版本
- 插件安装时校验契约版本兼容性

## 13.3 不要把业务控制权再次放回插件

即便插件只做算法，也不要允许插件：

- 自定义宿主菜单
- 自定义鉴权策略
- 自定义控制器路由

这些能力若未来需要，也必须采用**显式审批 + 白名单配置**模式接入。

## 14. 最终建议

**建议正式将 IotPlatform 插件体系收敛为“能力型插件架构”。**

对当前场景，推荐优先实施以下落地顺序：

1. 在 `Plugin.Contract` 增加 `IApsSchedulingProvider` 与相关 DTO
2. 在主项目新增固定 APS 接口 `POST /aps/scheduling/run`
3. 改造 `InjectionApsCpSatDemo`，移除 `ApsSchedulingAppService`
4. 在 `PluginManager` 中移除插件动态路由发现与分发逻辑
5. 增加能力白名单和插件 ID 白名单
6. 完成端到端安装、启用、算法调用、禁用、卸载验证
