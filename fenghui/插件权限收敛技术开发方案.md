# 通用插件权限收敛与受控调用技术开发方案

## 1. 背景与结论

当前插件模块允许插件程序集内直接声明动态接口，插件安装并启用后即可把接口挂到宿主请求管道中。  
这种模式对“快速扩展”友好，但对通用插件平台并不理想，主要问题是：

1. **权限过大**：插件拥有直接扩展宿主 HTTP API 的能力。
2. **安全边界不清**：宿主难以统一鉴权、审计、限流、超时和参数治理。
3. **接口冲突风险高**：插件与插件、插件与宿主都可能发生路由冲突。
4. **治理困难**：外部调用者无法清晰区分“平台接口”与“插件自带接口”。
5. **平台定位偏移**：当前插件模块定位应是**通用插件平台**，而不是“允许任意插件扩展 Web API”的动态接口平台。

**结论**：应将插件体系从“动态接口型插件”调整为“受控执行型通用插件”。  
也就是：

- **插件只负责实现能力 / 算法 / 规则 / 处理逻辑**
- **对外固定接口由宿主统一提供**
- **宿主通过 `pluginId` 定位并调用目标插件**
- **宿主只处理最小信封协议，不介入插件业务 JSON 内部结构**

这比原方案更适合作为 IotPlatform 的长期插件架构。

---

## 2. 方案目标

### 2.1 总体目标

将现有模型：

> 插件 = 元数据 + 生命周期 + DI + 动态接口

改造成：

> 插件 = 元数据 + 生命周期 + DI + 受控执行器

### 2.2 宿主职责

- 提供固定调用入口
- 做统一鉴权
- 做统一审计
- 做统一限流、超时、异常包装
- 校验 `pluginId`
- 校验插件状态
- 选择并调用目标插件
- 记录调用日志和执行结果

### 2.3 插件职责

- 实现受控执行接口
- 自己解析插件业务 JSON
- 自己处理插件内部业务规则
- 返回统一结果信封

### 2.4 核心原则

1. **插件不得直接暴露 HTTP 路由**
2. **宿主只认统一调用协议，不认插件自定义控制器**
3. **宿主只校验外层信封，不解析插件业务载荷**
4. **插件内部模型由插件自己定义，避免宿主契约膨胀**
5. **宿主通过设计模式实现插件定位与执行，提升稳健性与扩展性**

---

## 3. 为什么本方案优于原方案

| 维度 | 原方案：插件直接出接口 | 新方案：宿主固定入口 + pluginId 受控调用 |
|---|---|---|
| 安全性 | 低 | 高 |
| 路由冲突 | 高 | 极低 |
| 平台治理 | 弱 | 强 |
| 鉴权一致性 | 差 | 好 |
| 审计追踪 | 难 | 容易 |
| 插件自由度 | 高但失控 | 适中且可控 |
| 契约复杂度 | 表面低，实则分散 | 外层统一、内层自治 |
| 通用性 | 偏“动态 Web 扩展” | 真正通用插件平台 |

**结论**：如果插件模块定位是“通用插件平台”，应优先采用新方案。

## 4. 通用插件调用模型

## 4.1 最小信封协议

基于你的补充说明，宿主与插件之间不再继续扩张强类型业务 DTO，改为采用**最小固定外层协议 + 插件自定义内层 JSON**。

推荐统一请求模型：

```json
{
  "pluginId": "com.iotplatform.plugin.aps-scheduling",
  "operation": "schedule.run",
  "requestId": "8d16f01a-3bcd-4e4c-9f55-9b7f78f8d8d7",
  "schemaVersion": "1.0",
  "context": {
    "tenantId": "1001",
    "userId": "2001"
  },
  "payload": {
    "jobs": [],
    "machines": [],
    "settings": {}
  }
}
```

推荐统一响应模型：

```json
{
  "succeeded": true,
  "code": "OK",
  "message": "success",
  "requestId": "8d16f01a-3bcd-4e4c-9f55-9b7f78f8d8d7",
  "data": {
    "schedule": []
  },
  "meta": {
    "elapsedMs": 325
  }
}
```

### 说明

- `pluginId`：宿主定位目标插件
- `operation`：插件内部动作标识，避免一个插件只有单一行为
- `requestId`：便于追踪、幂等和日志关联
- `schemaVersion`：便于插件侧演进自身 JSON 协议
- `context`：宿主透传的调用上下文
- `payload`：插件自定义 JSON，宿主不介入其内部结构

## 4.2 对“只传 pluginId + 插件 JSON 对象”的评估

你的初步想法**可行**，但建议在最小协议上再补齐 `operation` 和 `schemaVersion` 两个字段。

原因：

1. **没有 `operation`，插件很快会退化成“大一统入口 + 大量 if/else”**
2. **没有 `schemaVersion`，插件协议升级后难以兼容**
3. **只有 `pluginId + payload`，日志可观测性和故障定位不够**

因此推荐采用：

> `pluginId + operation + payload + 少量治理字段`

这仍然是**轻契约**，但比纯 `pluginId + payload` 更稳健。

## 5. 插件平台目标架构

### 5.1 总体调用链

```text
客户端
  -> 宿主固定接口 /plugin/execute
      -> 鉴权 / 限流 / 审计 / 外层信封校验
          -> 根据 pluginId 解析目标插件
              -> 根据 operation 解析目标执行器
                  -> 透传 payload 给插件
                      -> 插件内部解析 JSON 并执行业务
                          -> 返回统一结果信封
```

### 5.2 插件能力定位

插件不再是“接口扩展包”，而是：

- 算法插件
- 规则插件
- 流程计算插件
- 数据处理插件
- 协议处理插件
- 转换器插件

即：**通用执行型插件**。

## 6. 设计模式建议

宿主通过 `pluginId` 调用指定插件，建议采用以下模式组合。

## 6.1 Registry Pattern（注册表模式）

### 作用

维护当前已启用插件及其执行器元数据。

### 建议组件

- `PluginRuntimeRegistry`

职责：

- 缓存已启用插件
- 缓存插件支持的 `operation`
- 缓存插件执行器类型

示意：

```csharp
public interface IPluginRuntimeRegistry
{
    bool TryGet(string pluginId, out PluginRuntimeDescriptor? descriptor);
}
```

## 6.2 Strategy Pattern（策略模式）

### 作用

不同插件、不同 `operation` 的执行逻辑天然就是不同策略。

### 建议接口

```csharp
public interface IPluginExecutor
{
    bool CanHandle(string operation);
    Task<PluginInvokeResult> ExecuteAsync(
        PluginInvokeRequest request,
        PluginExecutionContext context,
        CancellationToken cancellationToken = default);
}
```

每个插件内部可以有一个或多个执行器：

- `ApsSchedulingExecutor`
- `RuleValidateExecutor`
- `DataTransformExecutor`

## 6.3 Factory Pattern（工厂模式）

### 作用

根据 `pluginId` 和 `operation` 从插件专属容器中解析出正确执行器。

### 建议组件

- `PluginExecutorFactory`

职责：

1. 获取插件运行时条目
2. 获取插件容器
3. 解析该插件下的执行器集合
4. 按 `operation` 选择合适执行器

## 6.4 Facade + Template Method（门面 + 模板方法）

### 作用

统一宿主调用流程，避免每个控制器重复写鉴权、日志、异常处理。

### 建议组件

- `PluginInvokeFacadeService`

标准流程：

1. 校验请求外层信封
2. 校验 `pluginId`
3. 校验插件启用状态
4. 校验插件是否允许执行
5. 通过工厂解析执行器
6. 包装执行上下文
7. 调用执行器
8. 统一记录日志
9. 统一包装结果

这部分非常适合用模板方法固化流程。

## 6.5 Chain of Responsibility（可选）

如需增强治理能力，可增加调用前置责任链：

- `PluginInvokeAuthHandler`
- `PluginInvokeRateLimitHandler`
- `PluginInvokeAuditHandler`
- `PluginInvokePayloadSizeHandler`

此模式不是首期必须，但适合二期演进。

## 8. 契约层改造方案

## 8.1 契约目标

契约层不再按具体业务持续膨胀，而是收敛为**通用调用契约**。

### 推荐保留的核心接口

```csharp
public interface IPlugin
{
    string Id { get; }
    string Name { get; }
    string Version { get; }
    PluginType Type { get; }
}
```

```csharp
public interface IPluginLifecycle
{
    Task StartAsync(PluginContext context);
    Task StopAsync(PluginContext context);
}
```

### 推荐新增的核心接口

```csharp
public interface IPluginExecutor
{
    bool CanHandle(string operation);
    Task<PluginInvokeResult> ExecuteAsync(
        PluginInvokeRequest request,
        PluginExecutionContext context,
        CancellationToken cancellationToken = default);
}
```

```csharp
public interface IPluginDescriptor
{
    IReadOnlyCollection<string> Operations { get; }
}
```

## 8.2 通用契约 DTO

建议在 `Plugin.Contract` 中只增加以下通用 DTO：

- `PluginInvokeRequest`
- `PluginInvokeResult`
- `PluginExecutionContext`
- `PluginOperationDescriptor`（可选）

建议目录：

```text
Plugin.Contract
├── Dto
│   ├── PluginInvokeRequest.cs
│   ├── PluginInvokeResult.cs
│   ├── PluginExecutionContext.cs
│   └── PluginOperationDescriptor.cs
├── IPlugin.cs
├── IPluginLifecycle.cs
├── IPluginExecutor.cs
└── IPluginDescriptor.cs
```

## 8.3 DTO 建议定义

```csharp
public sealed class PluginInvokeRequest
{
    public string PluginId { get; set; } = default!;
    public string Operation { get; set; } = default!;
    public string? RequestId { get; set; }
    public string? SchemaVersion { get; set; }
    public JsonObject? Context { get; set; }
    public JsonNode? Payload { get; set; }
}
```

```csharp
public sealed class PluginInvokeResult
{
    public bool Succeeded { get; set; }
    public string Code { get; set; } = "OK";
    public string? Message { get; set; }
    public string? RequestId { get; set; }
    public JsonNode? Data { get; set; }
    public JsonObject? Meta { get; set; }
}
```

### 结果

这样宿主与插件之间的契约就只剩一层很薄的通用信封，避免继续围绕每个业务场景扩张强类型接口。

## 9. PluginManager 改造方案

## 9.1 角色调整

`PluginManager` 不再承担“插件路由发现/分发器”角色，而回归“插件运行时管理器”。

## 9.2 核心职责

1. 插件扫描与安装
2. 插件启停与卸载
3. 插件专属 `ServiceProvider` 管理
4. 插件元数据缓存
5. 插件执行器解析

## 9.3 新增能力

### 1. 获取插件执行器

```csharp
public IReadOnlyList<IPluginExecutor> GetPluginExecutors(string pluginId)
```

或：

```csharp
public IPluginExecutor? GetPluginExecutor(string pluginId, string operation)
```

### 2. 获取插件描述信息

```csharp
public PluginRuntimeDescriptor? GetDescriptor(string pluginId)
```

### 3. 运行时元数据缓存

建议 `PluginEntry` 增加：

- `Operations`
- `ExecutorTypes`

## 9.4 启用插件时的校验链

1. 解析 `plugin.json`
2. 加载程序集
3. 校验 `startupClass`
4. 创建插件容器
5. 解析 `IPluginDescriptor`
6. 解析 `IPluginExecutor`
7. 校验 `operation` 声明与执行器能力一致
8. 缓存到运行时注册表
9. 更新状态为 `Enabled`

## 9.5 失败处理

任一环节失败：

- 状态置为 `Error`
- 禁止进入可调用状态
- 输出明确日志

## 10. 宿主固定接口设计

## 10.1 首选方案：单一通用入口

建议宿主提供固定接口：

```text
POST /plugin/execute
```

特点：

- URL 固定
- 宿主只处理外层协议
- 插件通过 `pluginId + operation` 分流

这是最符合“通用插件平台”定位的方案。

## 10.2 可选方案：通用入口 + 领域门面

若未来某些业务想保留更清晰的领域接口，可在通用调用之上再增加宿主门面接口，例如：

- `POST /aps/schedule/run`
- `POST /rule/engine/execute`

但这些门面接口内部仍然调用统一的 `PluginInvokeFacadeService`，而不是直接调用插件控制器。

**建议首期优先实现单一通用入口**，降低系统复杂度。

## 10.3 宿主接口职责边界

宿主应负责：

1. 校验 `pluginId`
2. 校验 `operation`
3. 校验 payload 大小
4. 校验插件是否启用
5. 统一鉴权
6. 统一审计
7. 统一异常包装

宿主不负责：

- 解析插件业务 JSON 字段
- 理解插件内部算法语义
- 维护插件私有业务 DTO

## 12. 配置与授权建议

## 12.1 动态接口总开关是否保留

**不需要保留。**

原因是本方案已经明确把“插件动态暴露 HTTP 接口”定义为将被移除的旧能力，而不是一个长期受支持的可选特性。  
既然目标状态是彻底取消该能力，就不应再保留 `EnableDynamicApiFromPlugin` 这类兼容性开关，否则会带来以下问题：

1. 容易让旧模型长期残留，影响架构收敛。
2. 增加代码分支和测试复杂度。
3. 让安全边界重新变得可配置、不可预测。

因此本方案建议：

- **直接从技术方案中移除 `EnableDynamicApiFromPlugin`**
- 后续实现中也不再新增该配置项
- 旧动态接口能力按迁移计划下线，而不是通过开关长期并存

## 12.2 插件执行白名单统一开关

建议增加一个统一白名单开关，而不是为插件白名单和 operation 白名单分别建开关。

推荐配置：

```json
{
  "Plugin": {
    "EnableInvokeWhitelist": false,
    "AllowedPlugins": [
      "com.iotplatform.plugin.aps-scheduling"
    ],
    "AllowedOperations": {
      "com.iotplatform.plugin.aps-scheduling": [
        "schedule.run",
        "schedule.preview"
      ]
    }
  }
}
```

### 白名单生效规则

1. `EnableInvokeWhitelist = true` 时：
   - `AllowedPlugins`
   - `AllowedOperations`
   才参与校验。

2. `EnableInvokeWhitelist = false` 时：
   - 白名单配置整体不生效
   - 宿主只做基础校验，不做白名单拦截

3. 配置文件中**没有** `EnableInvokeWhitelist`、`AllowedPlugins`、`AllowedOperations` 时：
   - 必须兼容
   - 不得报错
   - 视为**白名单关闭**

### 实现要求

- 读取配置时必须提供安全默认值
- 不允许因缺失白名单配置导致启动失败
- 不允许因缺失白名单配置导致插件调用抛出配置异常

## 12.3 插件执行白名单

在 `EnableInvokeWhitelist = true` 时，建议校验插件白名单：

```json
{
  "Plugin": {
    "AllowedPlugins": [
      "com.iotplatform.plugin.aps-scheduling"
    ]
  }
}
```

## 12.4 Operation 白名单

在 `EnableInvokeWhitelist = true` 时，建议按插件维护允许调用的操作：

```json
{
  "Plugin": {
    "AllowedOperations": {
      "com.iotplatform.plugin.aps-scheduling": [
        "schedule.run",
        "schedule.preview"
      ]
    }
  }
}
```

这样可以避免插件在开启白名单模式后暴露额外行为。

`plugin.json` 仍保留为插件发现入口，但职责调整为：

- 标识插件
- 标识主程序集和启动类
- 声明插件支持的操作

```json
{

  "name": "注塑生产排程",
  "version": "1.0.0",
  "type": "BackgroundService",

  "startupClass": "Plugin.ApsScheduling.PluginStartup",
  "operations": [
    "schedule.run",

### 建议

- `operations` 用于宿主快速识别插件能力边界
- 启用时校验 `plugin.json` 与程序集内 `IPluginDescriptor` 是否一致

Executors\ApsSchedulingExecutor.cs
```

示例：

```csharp
public sealed class ApsSchedulingExecutor : IPluginExecutor
{
    public bool CanHandle(string operation) =>
        string.Equals(operation, "schedule.run", StringComparison.OrdinalIgnoreCase);

    public async Task<PluginInvokeResult> ExecuteAsync(
        PluginInvokeRequest request,
        PluginExecutionContext context,
        CancellationToken cancellationToken = default)
    {
        var payload = request.Payload?.Deserialize<SchedulingPayload>()
            ?? throw new InvalidOperationException("payload 不能为空");

        // 插件内部自行解析 payload，自行执行业务
        // 宿主不需要知道 SchedulingPayload 的具体结构

        var result = new
        {
            schedule = new object[0]
        };

        return new PluginInvokeResult
        {
            Succeeded = true,
            Code = "OK",
            Message = "success",
            RequestId = request.RequestId,
            Data = JsonSerializer.SerializeToNode(result)
        };
    }
}
```

## 14.5 插件内部 DTO 处理建议

对该插件，不建议再把输入输出模型放到宿主契约层。  
更推荐：

- 插件内部自定义 `SchedulingPayload`
- 插件内部自定义结果模型
- 通过 `JsonSerializer` 在插件内部完成 JSON 反序列化与结果序列化

这样宿主无需知道 APS 排程领域对象。

## 14.6 生产插件改造要求

由于 `InjectionApsCpSatDemo` 是生产插件，改造时必须额外满足以下要求：

1. 插件执行器的入参校验必须完整，不能只做演示级空判断。
2. 插件内部 JSON 解析失败时，必须输出明确异常和关键上下文。
3. 插件执行超时、中断、求解失败、依赖缺失等错误必须有清晰错误码和错误消息。
4. 插件输出结果结构必须保持向后兼容，变更时通过 `schemaVersion` 演进。
5. 插件日志必须能支撑生产问题排查，不能只输出简略错误描述。

## 15. 日志与异常处理规范

本方案要求日志体系按生产标准建设，尤其是异常日志必须足够完整。

## 15.1 宿主日志要求

宿主在以下关键节点必须记录日志：

1. 收到插件调用请求
2. 解析 `pluginId`
3. 解析 `operation`
4. 白名单校验
5. 插件执行开始
6. 插件执行结束
7. 插件执行失败

### 宿主成功日志建议字段

- `requestId`
- `pluginId`
- `operation`
- `tenantId`
- `userId`
- `elapsedMs`
- `resultCode`

### 宿主异常日志必须包含

- 完整异常堆栈（stack trace）
- 根因异常消息
- `pluginId`
- `operation`
- `requestId`
- 关键上下文参数
- 插件状态
- 是否命中白名单校验

## 15.2 插件日志要求

插件内部至少要记录：

1. 执行器开始执行
2. 关键参数摘要
3. 核心阶段耗时
4. 执行结果摘要
5. 异常详情

### 插件异常日志必须包含

- 完整堆栈
- 原始错误原因
- `operation`
- `requestId`
- `schemaVersion`
- 关键业务参数摘要
- 关键依赖状态

### 关键参数日志要求

“关键参数”不要求把整个 `payload` 原文全部打印到日志，但应至少输出：

- 数据规模摘要（如工单数、设备数、规则数）
- 关键标识（租户、插件、请求号）
- 关键模式参数（如算法模式、求解时间上限）

对于体积过大或敏感数据：

- 应输出摘要或脱敏后的内容
- 不建议无上限打印整个大 JSON

## 15.3 异常包装建议

推荐分两层处理：

1. **插件内部**
   - 对可预期业务异常进行业务化包装
   - 保留原始异常作为 InnerException

2. **宿主外层**
   - 统一记录详细异常日志
   - 统一返回标准 `PluginInvokeResult`

### 推荐原则

- 日志里保留详细堆栈
- 对外返回可控错误码和错误消息
- 不吞异常
- 不只记录 `ex.Message`

## 15.4 日志示例要求

宿主错误日志建议达到以下粒度：

```text
插件调用失败
requestId=xxx
pluginId=com.iotplatform.plugin.aps-scheduling
operation=schedule.run
pluginState=Enabled
whitelistEnabled=true
tenantId=1001
userId=2001
payloadSummary={jobCount:120,machineCount:15}
reason=Google.OrTools solver timeout
stackTrace=...
```

这类信息应被视为必需项，而不是可选项。

## 17. 验收标准

### 17.1 主项目

1. 插件启用后，不再自动新增 HTTP 路由
2. 宿主只有固定入口可调用插件
3. 宿主可通过 `pluginId + operation` 正确定位并调用插件
4. 未启用插件无法被调用
5. 当 `EnableInvokeWhitelist = true` 时，非白名单插件或 operation 无法被调用
6. 当白名单配置缺失时，系统仍能正常启动和调用
7. 插件异常能被统一捕获并记录完整堆栈、错误原因及关键参数

### 17.2 插件项目

1. `InjectionApsCpSatDemo` 中不再出现 `IDynamicApiController`
2. 插件仍可安装、启用、禁用、卸载
3. 插件能通过宿主固定入口正常执行
4. 插件协议升级可通过 `schemaVersion` 演进
5. 插件日志满足生产问题排查要求

## 18. 最终建议

建议正式把当前插件体系调整为：

> **通用插件平台 + 统一调用信封 + 受控执行器架构**

落地顺序建议如下：

1. 在 `Plugin.Contract` 中建立最小通用执行契约
2. 在宿主实现固定入口 `/plugin/execute`
3. 在插件模块引入 **Registry + Strategy + Factory + Facade** 组合模式
4. 把 `InjectionApsCpSatDemo` 改造成生产级执行器插件
5. 加入统一白名单开关与兼容缺省配置逻辑
6. 建立宿主与插件两侧的详细日志规范
7. 逐步下线插件动态接口能力
