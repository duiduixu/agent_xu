# 插件开发说明文档

## 1. 模块位置与结构

插件管理模块由 3 个类库和 1 个宿主接入点组成：

- `02-应用模块\15-Plugin\Plugin`：插件运行时与管理接口
- `02-应用模块\15-Plugin\Plugin.Interface`：插件开发契约
- `02-应用模块\15-Plugin\Plugin.Entity`：插件实体、枚举、DTO
- `03-应用服务\IotPlatform.Web.Core`：宿主配置与启动加载

当前目录结构已按本项目其他应用模块的常见方式保持为：

- `Plugin` 负责运行时逻辑
- `Plugin.Interface` 只放接口契约
- `Plugin.Entity` 只放实体/枚举/DTO

其中运行时专用类型 `PluginEntry` 已保留在 `Plugin` 项目中，不再放入 `Plugin.Entity`，避免数据契约层混入运行时状态。

---

## 2. 当前能力

当前插件模块支持：

1. 插件目录扫描
2. ZIP 插件安装
3. 插件启用 / 禁用 / 卸载
4. 独立 `AssemblyLoadContext` 加载与卸载
5. 插件内部 DI 注册
6. 后台生命周期回调
7. 插件状态写入 `system_plugin` 表

当前优先支持的插件类型：

- `BackgroundService`

预留但未完整实现的类型：

- `ChannelProtocol`
- `PageComponent`

---

## 3. 插件项目必须引用的契约

插件至少需要引用：

- `Plugin.Interface`

如果需要直接使用插件枚举或上下文对象，可同时引用：

- `Plugin.Entity`

---

## 4. 插件入口接口

所有插件必须实现 `Plugin.Interface.IPlugin`：

```csharp
using Furion;
using Plugin.Entity.Enum;
using Plugin.Interface;

namespace DemoPlugin;

public sealed class DemoPluginStartup : AppStartup, IPlugin
{
    public string Id => "com.iotplatform.plugin.demo";
    public string Name => "演示插件";
    public string Version => "1.0.0";
    public PluginType Type => PluginType.BackgroundService;
}
```

字段要求：

| 字段 | 说明 |
| --- | --- |
| `Id` | 全局唯一，建议使用反向域名格式 |
| `Name` | 插件显示名称 |
| `Version` | 语义化版本号 |
| `Type` | 当前建议固定为 `BackgroundService` |

---

## 5. 可选生命周期接口

如果插件需要在启用后持续运行，应额外实现 `IPluginLifecycle`：

```csharp
using Plugin.Entity.Dto;
using Plugin.Interface;

public sealed class DemoPluginStartup : AppStartup, IPlugin, IPluginLifecycle
{
    public string Id => "com.iotplatform.plugin.demo";
    public string Name => "演示插件";
    public string Version => "1.0.0";
    public PluginType Type => PluginType.BackgroundService;

    public Task StartAsync(PluginContext context)
    {
        context.Logger?.LogInformation("插件启动：{Dir}", context.PluginDirectory);
        return Task.CompletedTask;
    }

    public Task StopAsync(PluginContext context)
    {
        context.Logger?.LogInformation("插件停止");
        return Task.CompletedTask;
    }
}
```

`PluginContext` 提供：

- `PluginDirectory`：插件物理目录
- `Configuration`：数据库中的插件配置 JSON
- `Logger`：插件专用日志

---

## 6. 插件内部依赖注入

如果插件内部需要注册服务，插件入口类应继承 `Furion.AppStartup`，并提供：

```csharp
using Microsoft.Extensions.DependencyInjection;

public void ConfigureServices(IServiceCollection services)
{
    services.AddSingleton<IMyService, MyService>();
}
```

宿主在启用插件时会：

1. 创建插件子容器
2. 调用插件 `ConfigureServices`
3. 为插件构建独立 `ServiceProvider`

---

## 7. plugin.json 格式

每个插件包必须包含 `plugin.json`：

```json
{
  "id": "com.iotplatform.plugin.demo",
  "name": "演示插件",
  "version": "1.0.0",
  "description": "一个示例后台插件",
  "type": "BackgroundService",
  "assembly": "DemoPlugin.dll",
  "startupClass": "DemoPlugin.DemoPluginStartup"
}
```

字段说明：

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `id` | 是 | 插件唯一标识 |
| `name` | 是 | 插件名称 |
| `version` | 是 | 插件版本 |
| `description` | 否 | 插件描述 |
| `type` | 是 | 当前建议使用 `BackgroundService` |
| `assembly` | 是 | 主程序集文件名，必须是 `.dll` |
| `startupClass` | 是 | 入口类完整类型名 |

---

## 8. 插件打包要求

ZIP 包中至少应包含：

```text
DemoPlugin.zip
├─ plugin.json
├─ DemoPlugin.dll
├─ DemoPlugin.deps.json
├─ DemoPlugin.runtimeconfig.json
└─ 其它依赖 DLL
```

注意：

1. `plugin.json` 必须位于 ZIP 内
2. `assembly` 指向的 DLL 必须真实存在
3. 依赖 DLL 需要随包提供
4. 不允许包含越界路径

---

## 9. 宿主配置

宿主配置文件：

- `03-应用服务\IotPlatform.Web.Core\Configuration\Plugin.json`

当前配置结构：

```json
{
  "Plugin": {
    "ScanPath": "Plugins",
    "PublicKey": "",
    "AutoEnable": true,
    "IsolationLevel": "AssemblyLoadContext"
  }
}
```

配置项说明：

| 配置项 | 说明 |
| --- | --- |
| `ScanPath` | 插件扫描目录 |
| `PublicKey` | 插件签名校验公钥，当前预留 |
| `AutoEnable` | 启动后是否自动启用数据库中已启用插件 |
| `IsolationLevel` | 当前保留配置，实际使用 `AssemblyLoadContext` |

## 10. 插件管理接口

当前接口路由基于 `plugin`：

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| GET | `/api/plugin/page` | 分页查询插件 |
| GET | `/api/plugin/{id}` | 查询插件详情 |
| POST | `/api/plugin/install` | 上传 ZIP 安装 |
| POST | `/api/plugin/{id}/enable` | 启用插件 |
| POST | `/api/plugin/{id}/disable` | 禁用插件 |
| DELETE | `/api/plugin/{id}` | 卸载插件 |

## 11. 生命周期说明

### 11.1 安装

安装流程：

1. 校验 ZIP 可读
2. 解析 `plugin.json`
3. 校验主程序集存在
4. 检查内存与数据库中的插件 ID 冲突
5. 解压到插件目录
6. 写入 `system_plugin`
7. 注册到运行时内存字典

### 11.2 启用

启用流程：

1. 按插件 ID 加锁，避免重复启用
2. 创建独立 `PluginLoadContext`
3. 精确查找 `startupClass`
4. 实例化插件入口类
5. 调用 `ConfigureServices`
6. 如实现 `IPluginLifecycle` 则调用 `StartAsync`
7. 更新数据库状态

### 11.3 禁用

禁用流程：

1. 按插件 ID 加锁
2. 调用 `StopAsync`
3. 释放插件子容器
4. 卸载 `AssemblyLoadContext`
5. 更新数据库状态

### 11.4 卸载

卸载流程：

1. 按插件 ID 加锁
2. 如已启用则先禁用
3. 删除插件目录
4. 删除数据库记录
5. 从运行时内存移除

## 12. 当前版本已处理的关键坑点

本轮已特别修复以下问题：

1. **并发启停冲突**：同一插件的安装、启用、禁用、卸载已增加串行锁
2. **ZIP 路径穿越**：解压时已阻止写出插件目录
3. **安装回滚缺失**：入库失败时会清理已解压目录
4. **数据库残留记录重复安装**：安装前会同时检查内存和数据库
5. **绝对路由误用**：接口改为相对路由，避免路径漂移
6. **运行时类型分层不清**：`PluginEntry` 已放回运行时项目
7. **状态写入假成功**：安装/状态更新/删除失败不再静默吞掉

---

## 13. 开发建议

1. 插件入口类保持无状态，重资源对象优先注册到 DI
2. 后台插件必须在 `StopAsync` 中释放连接、定时器和订阅
3. 插件不要依赖宿主私有实现细节，优先依赖公开契约
4. 主程序集名称、类型全名、`plugin.json` 字段必须严格一致
5. 升级同一插件时，先禁用再卸载，再安装新包

## 14. 最小示例

```csharp
using Furion;
using Microsoft.Extensions.DependencyInjection;
using Plugin.Entity.Dto;
using Plugin.Entity.Enum;
using Plugin.Interface;

namespace DemoPlugin;

public sealed class DemoPluginStartup : AppStartup, IPlugin, IPluginLifecycle
{
    public string Id => "com.iotplatform.plugin.demo";
    public string Name => "演示插件";
    public string Version => "1.0.0";
    public PluginType Type => PluginType.BackgroundService;

    public void ConfigureServices(IServiceCollection services)
    {
        services.AddSingleton<DemoWorker>();
    }

    public Task StartAsync(PluginContext context)
    {
        context.Logger?.LogInformation("Demo 插件启动");
        return Task.CompletedTask;
    }

    public Task StopAsync(PluginContext context)
    {
        context.Logger?.LogInformation("Demo 插件停止");
        return Task.CompletedTask;
    }
}

public sealed class DemoWorker
{
}
```

## 15. 结论

当前插件模块已具备“安装、启停、卸载、隔离加载、状态管理”的基础可用能力，适合作为首版后台服务插件框架继续扩展。后续如果要支持协议插件、前端页面插件或插件签名校验，建议在现有结构上继续分层扩展，而不要把运行时状态和数据契约重新混放。
