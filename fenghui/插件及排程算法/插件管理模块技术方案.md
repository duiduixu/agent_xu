# 插件管理模块技术方案（简化版）

## 一、目标

为 IotPlatform 增加插件管理能力，**首期仅实现后台服务插件**，满足将 `InjectionApsCpSatDemo`（注塑生产计划排程）作为插件接入平台的需求。保留 `PluginType` 字段，为后续扩展通道协议、页面组件等类型预留空间。

## 三、核心设计

### 3.1 插件加载机制

采用 **`AssemblyLoadContext`** 实现程序集级别的热插拔：

```
应用启动
    │
    ▼
扫描 Plugins/ 目录
    │
    ├── 读取每个子目录的 plugin.json（插件清单）
    │
    ├── 查询数据库 sys_plugin 表，比对状态
    │
    └── 对 state=Enabled 的插件：
          ├── PluginLoadContext 加载程序集
          ├── 反射实例化 IPlugin 实现类
          ├── Furion 自动调用 AppStartup.ConfigureServices()（注册 DI）
          └── 若实现了 IPluginLifecycle → 调用 StartAsync() 启动后台工作
             否则 → 仅标记 Enabled，等待 API 调用
```

- **程序集隔离**：每个插件使用独立的 `AssemblyLoadContext`（`isCollectible: true`），禁用时可完整卸载。
- **框架共享**：Furion、ASP.NET Core、SqlSugar 等框架程序集回退到默认加载上下文，避免重复加载。
- **依赖解析**：第三方 NuGet 包（如 Google.OrTools）从插件本地目录优先加载。

### 3.2 插件清单（plugin.json）

每个插件目录下必须包含 `plugin.json`：

```json
{
  "id": "com.iotplatform.plugin.aps-scheduling",
  "name": "注塑生产排程插件",
  "version": "1.0.0",
  "description": "基于 Google OR-Tools CP-SAT 的注塑机生产计划排程",
  "type": "BackgroundService",
  "assembly": "Plugin.ApsScheduling.dll",
  "startupClass": "Plugin.ApsScheduling.PluginStartup"
}
```

字段说明：

| 字段 | 说明 |
|------|------|
| `id` | 插件唯一标识（反向域名格式） |
| `name` | 显示名称 |
| `version` | 语义化版本号 |
| `type` | 插件类型：`BackgroundService` / `ChannelProtocol` / `PageComponent` |
| `assembly` | 主程序集 DLL 文件名 |
| `startupClass` | 实现 `IPlugin` 的启动类全名 |

### 3.3 插件接口

在 `IotPlatform.Core` 中定义两个接口——**`IPlugin` 仅含元数据（必须实现）**，生命周期方法剥离到 **`IPluginLifecycle`（按需实现）**：

```csharp
// 01-架构核心/IotPlatform.Core/Plugins/IPlugin.cs

namespace IotPlatform.Core.Plugins;

/// <summary>
/// 插件元数据契约——所有插件必须实现（仅身份信息，无生命周期方法）
/// </summary>
public interface IPlugin
{
    string Id      { get; }
    string Name    { get; }
    string Version { get; }
    PluginType Type { get; }
}

/// <summary>
/// 插件生命周期契约——仅需要长时间运行（后台服务、长连接等）的插件才需实现。
/// 无状态、按需调用的插件无需实现此接口。
/// </summary>
public interface IPluginLifecycle
{
    /// <summary>启用时回调：启动后台工作（如连接 MQTT、启动采集循环）</summary>
    Task StartAsync(PluginContext context);

    /// <summary>禁用时回调：停止后台工作、释放连接</summary>
    Task StopAsync(PluginContext context);
}

public enum PluginType
{
    BackgroundService = 1,   // 后台服务（首期实现）
    ChannelProtocol   = 2,   // 通道协议（预留）
    PageComponent     = 3    // 页面组件（预留）
}

public class PluginContext
{
    public string PluginDirectory  { get; set; } = default!;
    public string? Configuration   { get; set; }
    public ILogger? Logger         { get; set; }
}
```

**PluginManager 在启用/禁用时的分支逻辑**：

```csharp
// 启用时
if (plugin is IPluginLifecycle lifecycler)
{
    await lifecycler.StartAsync(context);   // 长生命周期插件：启动后台工作
}
// 否则仅标记状态为 Enabled（无状态插件：DI 已注册完毕，等待调用即可）

// 禁用时
if (plugin is IPluginLifecycle lifecycler)
{
    await lifecycler.StopAsync(context);    // 长生命周期插件：停止后台工作
}
// 卸载程序集、回收资源
```

### 3.4 插件管理器（PluginManager）

```csharp
// IotPlatform.Web.Core/Plugins/PluginManager.cs

public sealed class PluginManager : ISingleton
{
    private readonly ConcurrentDictionary<string, PluginEntry> _plugins = new();
    private readonly IServiceProvider _serviceProvider;
    private readonly ILogger<PluginManager> _logger;

    // ─── 核心 API ───

    /// <summary>扫描 Plugins/ 目录，加载所有插件清单</summary>
    public Task ScanAsync(string pluginsDir);

    /// <summary>安装插件（从 .zip 解压 → 入库）</summary>
    public Task<PluginEntry> InstallAsync(Stream zipStream);

    /// <summary>启用：加载程序集 → Initialize → Start</summary>
    public Task EnableAsync(string pluginId);

    /// <summary>禁用：Stop → 卸载程序集</summary>
    public Task DisableAsync(string pluginId);

    /// <summary>卸载：Disable + 删除文件 + 清理数据库</summary>
    public Task UninstallAsync(string pluginId);

    /// <summary>获取已启用的插件列表</summary>
    public IReadOnlyList<PluginEntry> GetEnabledPlugins();
}

public class PluginEntry
{
    public IPlugin? Plugin         { get; set; }
    public PluginManifest Manifest { get; set; } = default!;
    public PluginState State       { get; set; }
    public AssemblyLoadContext? LoadContext { get; set; }
}

public enum PluginState { Installed, Enabled, Disabled, Error }
```

## 四、数据模型

### 4.1 数据库表（PostgreSQL）

仅一张核心表：

```sql
CREATE TABLE sys_plugin (
    id              BIGINT PRIMARY KEY,              -- 雪花 ID
    plugin_code     VARCHAR(128) NOT NULL UNIQUE,    -- 插件唯一标识
    plugin_name     VARCHAR(128) NOT NULL,           -- 显示名称
    plugin_version  VARCHAR(32)  NOT NULL,           -- 版本号
    plugin_type     INT          NOT NULL DEFAULT 1, -- 1=后台服务 2=通道协议 3=页面组件
    description     VARCHAR(512),                    -- 描述
    assembly_name   VARCHAR(256) NOT NULL,           -- 主 DLL 文件名
    startup_class   VARCHAR(256) NOT NULL,           -- 启动类全名
    plugin_dir      VARCHAR(512) NOT NULL,           -- 插件目录路径
    config_json     TEXT,                             -- 插件配置 JSON
    state           INT          NOT NULL DEFAULT 0, -- 0=已安装 1=已启用 2=已禁用 3=错误
    installed_at    TIMESTAMP    NOT NULL,
    enabled_at      TIMESTAMP,
    created_at      TIMESTAMP    NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMP
);

CREATE INDEX idx_plugin_type  ON sys_plugin(plugin_type);
CREATE INDEX idx_plugin_state ON sys_plugin(state);
```

### 7.3 排程 API（无状态计算型）

所有排程数据通过 API 请求体传入，计算完成后结果直接在响应中返回，不落库。

```csharp
// Api/ApsSchedulingAppService.cs
public class ApsSchedulingAppService : IDynamicApiController
{
    private readonly HeuristicScheduler _heuristic;
    private readonly CpSatOptimizer _optimizer;

    public ApsSchedulingAppService(
        HeuristicScheduler heuristic,
        CpSatOptimizer optimizer)
    {
        _heuristic = heuristic;
        _optimizer = optimizer;
    }

    /// <summary>
    /// 执行排程计算：输入任务、机台、物料库存、换线规则 → 返回排程结果
    /// </summary>
    [HttpPost("/api/aps/schedule")]
    public ScheduleResult RunScheduling([FromBody] SchedulingInput input)
    {
        // Phase 1: 贪心启发式 → 初始解
        var hint = _heuristic.Build(input.Jobs, input.Machines, input.MaterialStocks);

        // Phase 2: CP-SAT 优化（传入 hint 热启动）
        var result = _optimizer.Optimize(
            input.Jobs,
            input.Machines,
            input.MaterialStocks,
            input.Settings,
            input.SetupConfig,
            hint);

        return result;
    }
}

// 入参 DTO
public class SchedulingInput
{
    public List<Job> Jobs { get; set; }                          // 待排程任务列表
    public List<Machine> Machines { get; set; }                  // 可用机台列表
    public Dictionary<string, int> MaterialStocks { get; set; }  // 物料库存
    public SchedulingSettings Settings { get; set; }             // 调度参数（权重）
    public SetupConfig SetupConfig { get; set; }                 // 换线规则
}
```

### 7.4 数据流说明

排程插件完全无状态，不依赖任何数据库表：

```
POST /api/aps/schedule
  Body: { jobs, machines, materialStocks, settings, setupConfig }
        │
        ▼
  HeuristicScheduler.Build()       ← Phase 1: 贪心启发式
        │
        ▼
  CpSatOptimizer.Optimize(hint)    ← Phase 2: CP-SAT 优化
        │
        ▼
  Response: { operations, objectiveValue, status }
```

- **jobs / machines / materialStocks**：业务数据由调用方（前端或外部系统）提供。
- **settings / setupConfig**：排程参数和规则由调用方传入，不存储在插件中。
- 这种方式下，插件仅提供算法能力，数据持久化由主平台或其他模块负责，后续如需落库，可在主平台侧增加业务表来存储输入和结果。

## 十一、后续扩展预留

当前 `PluginType` 已预留了三种类型，后续可按需扩展：

- **ChannelProtocol（2）**：新增 `IChannelProtocolPlugin` 接口，在 `PluginManager.EnableAsync` 中按类型分支处理，将现有 PLC 驱动（Modbus、Siemens 等）迁移为插件。
- **PageComponent（3）**：新增 `IPageComponentPlugin` 接口 + 前端路由注册机制，在 `plugin.json` 中增加 `routes` 字段描述前端路由和菜单。
- **插件间通信**：可引入事件总线（项目已有 EventBus 基础设施），插件通过 `IEventBus` 与主程序或其他插件解耦通信。

这些扩展无需改动当前数据模型和核心接口——仅需在 `PluginType` 枚举和 `PluginManager` 增加对应分支即可。
